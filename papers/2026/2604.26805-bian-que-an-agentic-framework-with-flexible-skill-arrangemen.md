# Bian Que: An Agentic Framework with Flexible Skill Arrangement for Online System Operations

## Why this matters
Large-scale online systems (search, recommendation, ads) generate dozens of releases per day and hundreds of alerts. On-call engineers spend enormous time manually triaging: which metrics matter for this service? Which runbook applies here? What changed recently? BIAN QUE automates this triage loop by letting an LLM generate and maintain 'Skills' — structured documents that specify exactly which data to fetch and how to reason over it for each (business, module, scenario) combination. The practical payoff is dramatic: 75% fewer fired alerts, 95% reduction in pager noise, 80% root-cause accuracy, and >50% MTTR compression on a real production search engine serving hundreds of millions of users. For a solo developer, this is immediately useful as an on-call assistant for any system that emits metrics/logs/change events — you wire up your data sources once, describe your modules, and the LLM generates and self-improves the analysis playbooks automatically.

## Technical idea
The paper proposes three canonical O&M patterns (release interception, proactive inspection, alert root-cause analysis) each handled by a dedicated agent. The key innovation is 'Flexible Skill Arrangement': each Skill is a YAML/Markdown document with three fields — LoadDataSchema (a JSON spec of which data sources and knowledge-base entries to query), Prompt (a structured reasoning template), and Meta (tags for keyword matching). When an event arrives with no matching Skill, the LLM auto-generates one from a source descriptor and capability descriptor, then validates it by dry-running the LoadDataSchema (retrying up to 3 times). When a practitioner gives natural-language feedback, the framework routes it to either update the LoadDataSchema (wrong data fetched) or the Prompt (correct data, wrong reasoning). A unified feedback loop simultaneously distills case memories into a persistent knowledge base (KV + KKV + vector indices) and refines the Skills, so both co-evolve from a single correction signal. Models above ~30B parameters perform comparably; the repo ships with an OpenAI-compatible LLM client and a Flask service skeleton where all data/knowledge integration points are defined as interfaces with TODO stubs.

## Implementation plan
1. CLONE AND EXPLORE REPO: git clone https://github.com/benchen4395/BianQue_Assistant, read the directory structure, run 'pip install -r requirements.txt', and study run_demo.py end-to-end to understand the Parse→Search→Fetch→Prompt→LLM→Feedback pipeline.

2. CONFIGURE LLM BACKEND: Copy set_env.sh, fill in LLM_API_BASE, LLM_API_KEY, LLM_MODEL pointing at any OpenAI-compatible endpoint (e.g., local Ollama with qwen2.5:32b, or a cloud API). Run 'bash run_demo.sh' — the full pipeline should complete on mock data and print a JSON diagnosis.

3. DEFINE YOUR DATA LAKE: Edit src/app/framework/data/data_lake.yaml. Add one entry per data source you own (metrics API, log query, change event feed, etc.) with type, handler function name, description, context_keys, return_type. Keep descriptions precise — the LLM reads them to decide what to include in generated Skills.

4. IMPLEMENT DATA PROVIDERS: In src/app/framework/data/providers.py, write one Python function per data source defined in step 3. Signature must be (params: dict, context: dict) -> Any. Start with the simplest possible stub that returns realistic-looking data (even static JSON is fine for MVP). Register each function in data_provider_registry.py.

5. STUB THE KNOWLEDGE BASE: In src/app/framework/knowledge/get_knowlege.py implement get_knowledge(service_name, topk) -> str. For MVP, return a hardcoded string with 2-3 known failure patterns for your service. Later upgrade to a real vector store (Chroma, Redis, etc.).

6. IMPLEMENT REQUEST PARSER: In src/app/skill_explore/input_parser.py implement parse_warning() to extract business_id, scene_id, and context dict from your real alert payload format. For MVP, define a simple JSON schema you control and parse it directly.

7. WRITE A CAPABILITY DESCRIPTOR: Create a plain-text or YAML file that describes, in natural language, which data sources are relevant to which scenarios (e.g., 'for usability alerts, always fetch CPU and error-rate metrics; for coredump alerts, also fetch stack traces'). This seeds Skill generation. Store it and wire it into skill_create.py where 'capability descriptor' is referenced.

8. GENERATE YOUR FIRST SKILL: Craft a minimal alert event JSON that matches your business_id and scene_id. Drop it into demo_data/mock_alert_request.json and run run_demo.sh. The framework will call skill_create, which prompts the LLM to generate a Skill YAML under local_config/. Inspect the generated file — verify LoadDataSchema references your real data sources.

9. VALIDATE SKILL EXECUTION: Run the demo again. Trace the 4-step executor log (Parse→Search→Fetch→LLM). Check that data is actually fetched from your providers, the prompt is assembled correctly, and the LLM returns a structured JSON diagnosis. Fix any validation errors (the framework retries up to 3 times automatically).

10. WIRE UP FEEDBACK LOOP: After getting a diagnosis, call the feedback endpoint (POST /feedback or invoke skill_feedback.py directly) with a natural-language correction string, e.g., 'You missed the downstream dependency metric; also check service-B latency'. Verify the Skill file on disk is updated and the next run uses the revised LoadDataSchema or Prompt.

11. ADD SHORT-TERM CASE MEMORY: Implement a simple append-only JSON log (or Redis list) that stores each (event, diagnosis, feedback) triple. Before each LLM call, retrieve the 3 most recent cases for the same business_id+scene_id and prepend them to the system prompt as 'recent case context'. This is the short-term memory pathway.

12. ADD KNOWLEDGE DISTILLATION (LONG-TERM): After each feedback-corrected case, prompt the LLM with 'Given this case and correction, extract a reusable operational insight as a JSON object with keys: module, failure_pattern, metric_signals, resolution.' Append the result to a knowledge.json file. Feed the top-k retrieved entries (by keyword match on module name) into the Skill's LoadDataSchema knowledge queries.

13. STAND UP THE FLASK SERVICE: Run 'bash run_flask.sh'. POST a real alert payload to /warning. Confirm the end-to-end HTTP flow works. Add a /feedback POST route if not present.

14. OFFLINE EVALUATION: Collect 20-30 real historical alerts with known root causes (annotated by yourself or a colleague). Run each through the framework, compare LLM output to ground truth, compute pass@1 and pass@3. Aim for >60% pass@1 before going live.

15. ITERATE SKILLS WITH FEEDBACK: For each failure in step 14, provide one natural-language correction via the feedback interface. Re-run those cases. Track pass@k improvement across correction rounds to validate the self-evolving loop.

## Dependencies
Runtime: Python 3.9+, Flask, PyYAML, openai (pip package for OpenAI-compatible HTTP client). LLM: Any OpenAI-compatible endpoint; paper uses Qwen3.5-35B-FP8 on a single NVIDIA Tesla L20 (48GB VRAM). For local inference, a 35B model in FP8 needs ~35GB VRAM (one A100/L40S/L20) or use a cloud API. Minimum viable quality threshold is ~30B parameters — models below 9B show substantial degradation (pass@1 drops from 72% to 47% at 9B, catastrophic below that). Knowledge store for MVP: plain JSON files or Redis (KV + sorted sets for KKV + any ANN service like FAISS or Chroma for vector index). Data sources: whatever your system already exposes — Prometheus, Elasticsearch, internal APIs; no new monitoring infrastructure required. No GPU needed if using a cloud LLM API. No training required anywhere in the pipeline.

## Limitations / risks
1. LLM-generated Skills can produce invalid LoadDataSchema JSON that references non-existent data source types — the 3-retry validation loop catches most but not all; you may need to hand-fix edge cases. 2. Keyword-based Skill matching (business_id + scene_id tags) is brittle for novel event types not seen during generation — a new alert type with no matching Skill triggers generation from scratch each time until the Skill is saved. 3. The knowledge base grows unboundedly without the daily deduplication/pruning job described in the paper — you must implement this or risk contradictory entries degrading LLM reasoning over time. 4. 80% RCA accuracy (production result) means 1 in 5 diagnoses is wrong — never auto-execute remediation actions without human confirmation; the paper explicitly notes closed-loop remediation is out of scope. 5. Cold-start problem: for newly deployed services with no case history and no handbook entries, the knowledge base is empty and Skill generation may produce generic unhelpful prompts. 6. The framework is a workflow skeleton, not a full agent — context management, multi-turn conversation, and tool-call orchestration must be added separately. 7. All Skills are stored as local files on disk — no concurrency control; if multiple events fire simultaneously and both trigger Skill generation for the same (business, module), you may get a race condition writing the same file. 8. Sensitive operational data (metrics, logs, change events) is sent to the LLM API — ensure your LLM endpoint is on-premise or compliant with your data governance policy.

## Next steps
1. Replace keyword-based Skill matching with embedding-based similarity search over Skill descriptions — use sentence-transformers to embed event metadata and Skill tags, enabling fuzzy matching for novel event types. 2. Implement the daily knowledge consolidation job: prompt the LLM to deduplicate and merge knowledge entries with similar embeddings, flag contradictions, and prune stale entries older than N days. 3. Add closed-loop remediation for low-risk actions (e.g., auto-restart a crashed pod, scale up a service) gated by a confidence threshold in the diagnosis JSON — start with read-only actions (alert suppression, ticket creation) before write actions. 4. Build a multi-agent coordination layer for cascading failures: when the alert RCA agent's diagnosis implicates a downstream service, automatically spawn a second agent execution for that service and merge the two diagnoses. 5. Add a web UI for practitioner feedback — a simple React/Vue form that shows the LLM diagnosis alongside the raw data, with a text box for correction and a one-click 'update Skill' button. 6. Quantify the co-evolution compounding effect: run a controlled A/B test where one branch freezes the knowledge base and only updates Skills, and another branch does both, measuring pass@1 trajectory over 30+ days. 7. Extend to proactive inspection: add a cron-triggered agent that runs daily Skill-based health checks across all registered services without waiting for an alert, implementing the second line of defense from the paper. 8. Package the framework as a Docker image with a docker-compose stack (Flask + Redis + FAISS) so it can be deployed in one command on any infrastructure.
