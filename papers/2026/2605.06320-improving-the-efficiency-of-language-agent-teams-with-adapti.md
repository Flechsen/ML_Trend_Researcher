## Metadata

## Why this matters
LLM agent teams are increasingly used for complex tasks like software development, data analysis, and debugging. Existing approaches either use rigid pipelines (MetaGPT, ChatDev) that can't adapt when tasks evolve mid-execution, or fully decentralized teams that waste tokens on conflicts, overwrites, and redundant work. LATTE solves this by giving agents a shared, evolving task graph they collectively maintain — cutting token usage by ~50%, reducing wall-clock time, and slashing inter-agent conflicts (8x fewer overwrites vs decentralized). Concrete use cases: automated software engineering agents, multi-agent research assistants, parallel code debugging pipelines, and any scenario where you want a team of LLMs to divide-and-conquer work without thrashing each other.

## Technical idea
LATTE (Language Agent Teams for Task Evolution) coordinates a team of LLM agents through a shared dynamic coordination graph — a DAG where nodes are subtasks with statuses (pending/assigned/in_progress/done/verified), edges encode dependencies, and each node has an assigned agent. A Lead agent initializes the graph and holds exclusive rights to graph-wide operations (ASSIGN, VERIFY, CLOSE, RELEASE). Worker agents handle local operations (CLAIM a frontier task, COMPLETE their task, DISCOVER new subtasks they uncover during execution). The frontier F_t is the set of pending nodes whose dependencies are all done — this determines how many workers can run in parallel. Each round: (1) heartbeat monitoring flags stalled workers to Lead, (2) frontier is computed, (3) idle workers self-claim from frontier or Lead assigns them, (4) all selected agents execute in parallel, (5) graph is updated with emitted actions. The Lead sees only the graph (not full execution traces), workers see only their assigned subtask and direct predecessors — this context scoping prevents token bloat. The probabilistic framing treats graph evolution as approximate posterior inference (Metropolis-Hastings style): workers propose local updates from their scoped traces, Lead evaluates and accepts/rejects globally.

## Implementation plan
1. CLONE AND SETUP: Clone https://github.com/emieczkowski/latte, run 'pip install -r requirements.txt', create .env with ANTHROPIC_API_KEY or OPENAI_API_KEY, set LLM_PROVIDER and LLM_MODEL.

2. UNDERSTAND THE GRAPH DATA STRUCTURE: Read config.py and the orchestrator source. The core object is a DAG with nodes carrying {id, title, description, status, agent, dependencies}. Implement or confirm a GraphState class with: nodes dict, edges as adjacency list, a frontier() method that returns all pending nodes whose deps are all 'done', and apply_operation(op) that mutates state.

3. IMPLEMENT THE 7 GRAPH MUTATION OPERATORS: Write functions for DISCOVER(v, deps), ASSIGN(v, worker), CLAIM(v), COMPLETE(v), RELEASE(v), CLOSE(v), VERIFY(v) — each with precondition checks (e.g. CLAIM requires v in frontier and agent is None or self, COMPLETE requires status==in_progress and caller==assigned agent). These should raise on precondition failure and return the updated graph.

4. BUILD THE AGENT PROMPT TEMPLATES: Create two prompt templates — Lead prompt (sees full graph state as JSON/XML, knows all operators, responsible for ASSIGN/RELEASE/VERIFY/CLOSE/DISCOVER, must NOT edit files directly) and Worker prompt (sees only assigned subtask + direct predecessors, can CLAIM/COMPLETE/DISCOVER, can edit files and run scripts). Use the exact prompt text from Appendices A4.2 and A4.3 as starting points.

5. BUILD THE ACTION PARSER: Agents emit XML-style action tags in their text output. Write a parser that extracts: <discover_task id=... title=... dependencies=...>, <assign_task id=... to=...>, <claim_task id=...>, <complete_task id=...>, <release_task id=...>, <close_task id=...>, <verify_task id=...>, <broadcast>...</broadcast>, <edit_file path=...>...</edit_file>, <run_script path=.../>, <run_tests />, <read_file path=.../>. Map each to either a graph mutation or a file system action.

6. IMPLEMENT THE SANDBOX FILE ENVIRONMENT: Create a per-run working directory. Implement handlers for: edit_file (write content to path), read_file (return file contents), run_script (subprocess Python execution with stdout/stderr capture, timeout), run_tests (run pytest on the task's test suite, return results). All file ops are scoped to the run directory.

7. IMPLEMENT THE ORCHESTRATOR LOOP: Per-round execution: (a) Check heartbeats — any worker assigned/in_progress for H=4 rounds with no action gets flagged to Lead. (b) Compute frontier F_t. (c) Dispatch: Lead runs first if graph changed or heartbeat fired or Lead idle for H rounds. Then dispatch workers — in-progress workers continue, idle workers get assigned or can self-claim from frontier, max one worker per frontier node. (d) Run all selected agents in parallel (asyncio or threadpool), collect responses. (e) Parse actions from each response, apply graph mutations in order (FIFO for concurrent CLAIMs on same node). (f) Execute file system actions. (g) Check termination: all nodes done → return. Max rounds reached → return.

8. IMPLEMENT THE PLANNING PHASE: Before the execution loop, run Lead alone with the task description. Give Lead up to 5 turns to emit DISCOVER operations that seed the initial graph G_0. Lead context: task description + current graph state. Parse and apply all DISCOVER ops to build G_0.

9. WIRE UP A TASK: Use the included 'tasks/simple_eda/' task. Load task_description.json (fields: project, description, success_criteria, tasks). Run setup_data.py to stage the HR dataset. Then run the orchestrator. Evaluate by running test_suite.py on the output directory.

10. RUN THE MINIMAL EXPERIMENT: Execute 'python scripts/run_dynamic_pipeline.py --tasks tasks/simple_eda/task_description.json --n 4' and confirm it produces a runs/ directory with graph evolution logs, final outputs, and passes the test suite. Then run the same task with 'python scripts/run_no_graph_pipeline.py' to compare token usage and accuracy.

11. IMPLEMENT METRICS COLLECTION: Log per-run: total input+output tokens, wall-clock time, number of rounds, operator usage counts (DISCOVER/CLAIM/RELEASE/VERIFY calls), cross-agent overwrites (detect when agent B writes a file/function that agent A wrote in a prior round), concurrent writes (two agents writing same file in same round), wasted characters (chars written that don't appear in final output), inter-agent message count and total characters, agent idle rate (rounds where agent had no dispatch).

12. RUN THE DEBUG SPRINT TASK AND COMPARE: Run 'python scripts/run_dynamic_pipeline.py --tasks tasks/debug_sprint/task_description.json --n 4' vs the no_graph baseline. Observe straggler detection in action (RELEASE being invoked) and VERIFY operator on high-uncertainty nodes.

## Dependencies
Python 3.11+. Core LLM APIs: anthropic>=0.25 (claude-sonnet-4-6 or claude-3-5-sonnet) or openai>=1.30 (gpt-4o or gpt-4.1). The paper uses claude-sonnet-4-6 and gpt-5.2 specifically. Key Python packages from requirements.txt: asyncio (stdlib), subprocess (stdlib), pytest for test suite execution, python-dotenv for env management, pandas/numpy for EDA task data, matplotlib for visualization task artifacts, jupyter for analysis notebooks. Compute: No GPU required — all inference is via API. Cost estimate per full experiment replication: ~$250 for Anthropic (claude-sonnet-4-6, 150 trials) + ~$260 for OpenAI (gpt-5.2, 150 trials). For MVP single-task testing: budget ~$5-15 per run depending on model and task complexity. No local model weights needed. File system access required for sandbox (temp directories per run). The repo ships with pre-computed experiment CSVs so you can inspect results without re-running everything.

## Limitations / risks
1. PLANNING OVERHEAD ON SIMPLE TASKS: LATTE adds ~14.5s planning latency upfront. For tasks solvable in 1-2 rounds by a single agent, LATTE's graph overhead will dominate and make it slower/more expensive than just using one agent. The paper acknowledges this explicitly.

2. LLM ACTION PARSING FRAGILITY: Agents must emit correctly-formed XML action tags. Models sometimes hallucinate malformed tags, forget to emit COMPLETE, or emit conflicting operations. The orchestrator needs robust fallback handling for parse failures.

3. GRAPH MUTATION SERIALIZATION: Concurrent CLAIM conflicts are resolved FIFO by processing order, which is somewhat arbitrary. If two workers claim the same node, the loser must re-poll — this adds latency and requires workers to handle rejection gracefully.

4. LEAD AS BOTTLENECK: Despite being hybrid, the Lead still serializes all graph-wide mutations. If the Lead hallucinates a bad graph structure (wrong dependencies, premature CLOSE), it can block all workers. The paper notes Lead hallucination as a known failure mode.

5. CONTEXT WINDOW LIMITS: Workers cap at last 20 messages, Lead at last 10. Long-running tasks or large files can cause relevant context to fall off. edit_file contents embedded in context can blow up token counts quickly.

6. TASK DOMAIN LIMITATIONS: Current operator set is optimized for coding tasks with clear subtask boundaries. Open-ended reasoning tasks (essay writing, research synthesis) without natural decomposition points may not benefit as much.

7. API COST NONDETERMINISM: Parallel agent dispatch means costs scale with team size × rounds. A stuck team that hits the 40-round limit can burn 5-10x the tokens of a successful run.

8. MODEL VERSION DRIFT: The paper uses claude-sonnet-4-6 and gpt-5.2 specifically. Different model versions may follow action format instructions less reliably, changing coordination behavior significantly.

9. FILE CONFLICT EDGE CASES: The paper reduces but does not eliminate overwrites (still 4.3x per trial average). Simultaneous file edits in the same round can still corrupt state if both writes complete before conflict detection.

## Next steps
1. ADD A NEW CUSTOM TASK: Follow the README's 4-step process — create task_description.json, test_suite.py, optional setup_data.py, and run. Good candidate: a multi-file API server implementation where modules have clear dependencies.

2. IMPLEMENT STRAGGLER AUTO-RELEASE: Add the --straggler-rounds flag behavior so the orchestrator automatically calls RELEASE(v) after H rounds without progress, rather than just notifying the Lead. Measure impact on tail latency (paper shows 2.3x reduction at 95th percentile).

3. ADD LOCAL MODEL SUPPORT: Wrap the LLM calls to support Ollama or vLLM with llama-3.1-70B or Qwen-2.5-72B. The action-tag format is simple enough that capable open-source models should work, dramatically cutting cost.

4. BUILD A REAL-TIME GRAPH VISUALIZER: Use networkx + matplotlib or a web UI (Streamlit) to render the evolving DAG in real time during a run, coloring nodes by status. This makes LATTE's core interpretability advantage tangible.

5. EXTEND OPERATOR SET FOR OPEN-ENDED TASKS: Add a MERGE(v1, v2) operator to combine parallel subtask outputs, and a SPLIT(v, subtasks) operator for workers to decompose their own assigned node further. This would let LATTE handle less-structured tasks like literature review or data synthesis.

6. FINE-TUNE A SMALL COORDINATOR MODEL: Collect successful run traces (graph state → Lead action sequences) and fine-tune a small model (Qwen-2.5-7B or Llama-3.2-3B) to serve as the Lead, reducing Lead inference cost while keeping full-size workers.

7. IMPLEMENT REINFORCEMENT LEARNING ON COORDINATION OUTCOMES: Use trial success/failure and token efficiency as reward signal to learn better graph initialization policies for the Lead's planning phase — the paper explicitly flags this as future work.

8. BENCHMARK AGAINST AUTOGEN AND LANGGRAPH: The paper compares MetaGPT, Leader-Worker, and decentralized baselines but not AutoGen or LangGraph. Adding these comparisons would strengthen the empirical case and reveal where LATTE's graph structure uniquely helps.
