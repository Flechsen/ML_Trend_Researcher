# Pushing the Limits of LLM Tool Calling via Experiential Knowledge Integration and Activation

## Metadata
- arXiv ID: 2606.10875
- Authors: Yupu Hao, Zhuoran Jin, Huanxuan Liao, Kang Liu, Jun Zhao
- Published: 2026-06-09
- arXiv link: https://arxiv.org/abs/2606.10875v1
- PDF link: https://arxiv.org/pdf/2606.10875v1
- Code/repo: https://github.com/hypasd-art/KATE

## Why this matters
LLMs fail at multi-step tool calling not because they can't reason, but because they lack concrete experiential knowledge: what parameter constraints look like in practice, how to recover from errors, and which tool sequences work for specific scenarios. This matters for anyone building AI agents that call APIs, automate workflows, or interact with external systems. KATE shows you can get GPT-4.1-level tool-calling performance out of an 8B open model by systematically injecting retrieved execution trajectories and using parallel sampling to activate latent knowledge — without retraining if you don't want to.

## Technical idea
KATE is a three-stage framework. First, knowledge acquisition: extract ground-truth tool-call trajectories from a training split, optionally summarize them with an LLM into concise guidelines (Experience Summary), and encode them into a vector store using all-MiniLM-L6-v2. At inference time, when a user message arrives, retrieve the top-K most similar past trajectories by cosine similarity (threshold 0.5, filtered to the relevant toolset) and prepend them to the prompt. Second, knowledge activation: instead of just prompting the model harder (depth), run N=4 parallel samples at each interaction step with temperature=1, then aggregate via either self-consistency (majority vote) or LLM-based aggregation (feed all candidates back to the model to pick the best). Self-consistency is more stable; LLM aggregation can be better but degrades at high N. Third, optional knowledge internalization: fine-tune with knowledge-augmented data using RL (GRPO) which outperforms SFT alone, with LoRA rank=32. The key empirical findings are: instance-level trajectory knowledge beats abstract intent-level knowledge, parallel sampling beats prompt-hint engineering, and RL beats SFT for internalization.

## Implementation plan
1. ENVIRONMENT SETUP: Clone https://github.com/hypasd-art/KATE, install requirements.txt (Python 3.8+), set OPENAI_API_KEY for GPT-4o summarization calls, set up vLLM to serve Qwen3-8B locally (export BASE_URL=http://localhost:8000/v1/ and API_KEY=EMPTY), set CUDA_VISIBLE_DEVICES. Install all-MiniLM-L6-v2 from HuggingFace locally.

2. DATA ACQUISITION: Download BFCL-V3 benchmark data into bfcl_eval/data/. Split by sample ID: even IDs go to training set, odd IDs to test set. Pull 100 samples from Base scenario for training. This is your knowledge source and fine-tuning data.

3. BUILD KNOWLEDGE BASE (Scenario Trajectory): Run get_trajectory/extract_trajectory.py on the training split to extract ground-truth tool-call lists. This produces BFCL_experience.json with fields: sample_id, question, trajectory (the raw tool call sequence), tool_calls list.

4. BUILD KNOWLEDGE BASE (Experience Summary): Run get_trajectory/get_trajectory_summary.py with --model gpt-4o to have GPT-4o summarize each trajectory into concise operational guidelines. This produces BFCL_experience_summary.json. This step costs API credits but is optional — ST alone works well per ablation.

5. ADD EMBEDDINGS: Run get_trajectory/add_embedding.py with --model-path pointing to your local all-MiniLM-L6-v2. This encodes each user query into a dense vector and stores it alongside the trajectory. Output: BFCL_experience_with_embedding.json. This is the retrieval index used at inference.

6. IMPLEMENT RETRIEVAL LOGIC: At each inference step, when a user message arrives, encode it with all-MiniLM-L6-v2, filter the knowledge base to entries whose toolset intersects the current task's required tools (hard constraint), apply cosine similarity threshold p=0.5, sort descending, deduplicate, take top-K (K=3 is a reasonable start). Prepend retrieved examples to the prompt in the format shown in Prompt 7/8 in the paper appendix: 'Before answering... please first review the following related experiences: ### Example 1 **Question:** ... **Correct Tool Calling Trajectory:** ...' followed by a note that examples need not be followed if irrelevant.

7. IMPLEMENT PARALLEL SAMPLING: At each step of the multi-turn dialogue, instead of greedy decoding, sample N=4 candidate actions independently at temperature=1. If all N candidates are identical, use that action directly. Otherwise, apply self-consistency: majority vote over the N candidates (for structured tool calls, normalize the JSON before comparing). This is the most stable aggregation per Figure 3. For AppWorld code-generation tasks, use LLM-based aggregation instead (feed all candidates back to the model with Prompt 1/2 format).

8. WIRE UP THE MDP LOOP: Implement Algorithm 2 from the paper. At each turn: (a) observe user message, (b) retrieve relevant knowledge and prepend to history, (c) run N=4 parallel samples, (d) aggregate to get final action, (e) execute tool call and get environment feedback, (f) update history, (g) loop until terminal response or max steps T.

9. BASELINE EVALUATION: Run the FC baseline (no retrieval, no parallel sampling, temperature=0) on the BFCL-V3 test split across all four scenarios: Base, Miss Func, Miss Param, Long Context. Record scores. Expected baseline for Qwen3-8B: ~32-43 across scenarios per Table 1.

10. EVALUATE KATE INFERENCE-ONLY: Run with retrieval (ST knowledge) + parallel sampling (N=4, self-consistency) at temperature=1. Expected improvement: +10-16 points over FC baseline per Table 1. Compare w/o PS (retrieval only) and w/o Exp (parallel sampling only) ablations.

11. OPTIONAL — TRAINING STAGE (RL): Augment training data by concatenating retrieved ST knowledge with each training example's user instruction. Decompose multi-turn sequences into individual turn-level samples. Filter samples over 8192 tokens. Use GRPO: batch size 128, max prompt length 8192, max response length 2048, lr=3e-5, 8 sampled trajectories per prompt, 7 epochs, LoRA rank=32 alpha=16. Reward signal: exact match of tool call outputs against ground truth. This yields additional +2-4 points over inference-only KATE.

12. EVALUATE AND COMPARE: Run bfcl evaluate on all result directories. Compare FC, Memp baseline (ST+ES retrieval without parallel sampling, following Fang et al. 2025 proceduralization setup), KATE inference-only, and KATE+RL. Reproduce Table 1 numbers.

## Dependencies
Models: Qwen3-8B (primary, ~16GB VRAM for inference) and optionally Qwen3-32B (~65GB, needs multi-GPU). Served via vLLM. Retrieval: all-MiniLM-L6-v2 (sentence-transformers, ~90MB). Knowledge summarization: GPT-4o API (optional, only for ES variant). Evaluation: BFCL-V3 benchmark (public), AppWorld benchmark (public). Training: GRPO implementation (compatible with TRL or custom), LoRA via PEFT. Key Python packages: sentence-transformers, vllm, transformers, peft, trl, numpy, scikit-learn (for K-means in intent clustering), openai (for GPT-4o calls). Compute: minimum 1x A100 80GB or 2x A800 for Qwen3-8B inference with parallel sampling; 4x A100 recommended for RL training. Storage: ~50GB for model weights, ~1GB for knowledge base with embeddings. The repo is self-contained with bfcl CLI tooling for generate/evaluate steps.

## Limitations / risks
1. Parallel sampling multiplies inference cost by ~N=4x calls per step, though the paper shows token usage only increases ~1.3-1.9x in practice due to shorter trajectories from better accuracy — but latency does increase. 2. Self-consistency breaks down for tasks requiring diverse or structured outputs (e.g., code generation in AppWorld) where textual diversity makes majority voting unreliable — use LLM aggregation there but expect instability at high N. 3. On very hard tasks (AppWorld Test-Challenge), parallel sampling can introduce noise when the model's base capability is insufficient — all N samples may be wrong, and aggregating wrong answers produces a confidently wrong answer. 4. Knowledge base is static in these experiments; the paper notes that scaling the repository and making it dynamic remains unexplored. 5. Intent-level knowledge (SIC/TIC) requires LLM-based intent inference which adds latency and can introduce errors, and the paper shows it underperforms simple trajectory knowledge. 6. The training split is only 100 samples from Base scenario, which may not generalize broadly — the split-by-ID approach prevents leakage but limits training diversity. 7. RL training is sensitive to reward design; the matching-based evaluation used here may not capture partial correctness. 8. Results are reported as single runs (no error bars in main table, though Table 4 adds variance) so reported numbers may have run-to-run variance of ±1-3 points.

## Next steps
1. Dynamic knowledge base: implement online update where successful trajectories from deployment are automatically added to the retrieval store, enabling continual improvement without retraining. 2. Adaptive sampling: implement early stopping in parallel sampling — if top-2 candidates agree, skip remaining samples to reduce latency. 3. Confidence-weighted aggregation: instead of majority vote, use LLM-scored confidence per candidate and weight votes by confidence. 4. Scale the knowledge base: test with 1K, 10K trajectories to characterize the retrieval quality vs. knowledge base size curve the paper leaves unexplored. 5. Cross-domain transfer: build one knowledge base from BFCL and evaluate zero-shot on AppWorld to test transferability of experiential knowledge. 6. Multimodal extension: extend to vision-language tool calling where tool inputs include images (e.g., screenshot-based GUI agents). 7. Hierarchical retrieval: combine intent-level clustering (for coarse filtering) with instance-level similarity (for fine-grained ranking) to get the best of both knowledge types. 8. Reward shaping for RL: use partial credit rewards (e.g., correct tool name but wrong parameters gets 0.5) to improve sample efficiency in GRPO training. 9. Distillation: use KATE (large model) to generate high-quality augmented training data for a smaller model, enabling knowledge transfer without deploying the large model at inference time.
