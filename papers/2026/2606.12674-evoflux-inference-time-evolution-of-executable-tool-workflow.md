# Evoflux: Inference-Time Evolution of Executable Tool Workflows for Compact Agents

## Metadata
- arXiv ID: 2606.12674
- Authors: Kushal Raj Bhandari, Ling Yue, Ching-Yun Ko, Dhaval Patel, Shaowu Pan, Pin-Yu Chen, Jianxi Gao
- Published: 2026-06-10
- arXiv link: https://arxiv.org/abs/2606.12674v1
- PDF link: https://arxiv.org/pdf/2606.12674v1
- Code/repo: https://github.com/IBM/Evoflux

## Why this matters
Small language models (1.5B–4B parameters) are cheap to run locally or on-premise, but they fail badly at multi-step tool use: they pick unavailable tools, break dependency chains, emit malformed JSON, and hallucinate answers instead of grounding them in execution results. Evoflux shows that instead of trying to fix this by fine-tuning on a few hundred examples (which often makes things worse), you can get a 3%→17–24% execution feasibility improvement by running an evolutionary repair loop at inference time. Concrete use cases: (1) a local enterprise agent that calls internal APIs without sending data to a frontier model, (2) an edge-deployed assistant that orchestrates IoT or file-system tools, (3) a research scaffold for evaluating small planners on live tool catalogs without expensive retraining.

## Technical idea
Evoflux treats tool-use planning as a search problem over typed workflow DAGs rather than a one-shot generation problem. Each candidate is a typed graph: nodes are (server, tool, parameters) tuples, edges are data dependencies, and there is a terminal output node. An evolutionary loop (budget B iterations, population cap K) starts by having the small LLM generate an initial graph with up to R retries on failure. Each iteration: (1) compute adaptive exploration intensity I from an exponential moving average of recent score improvements G; (2) optionally trigger a meta-guidance redesign call if G stalls below threshold; (3) select a parent candidate (uniform if exploring, tournament if exploiting); (4) collect node-level execution evidence (which nodes failed, why, what schema errors occurred); (5) mutate via one of five typed edits—tool swap, parameter edit, tool insertion, tool removal, step reorder, or validator insertion—either randomly weighted by node error rate (exploration) or LLM-proposed given compressed failure context (exploitation); (6) compile, validate schemas, execute, score with a stability-tested LLM judge across six sub-dimensions; (7) prune the population using action-hash bucketing to maintain diversity. The final answer is the highest-scoring surviving candidate. Training baselines (SFT, SFT+DPO) use the same search-mined trajectories as supervision data but are shown to be unreliable under small-corpus budgets.

## Implementation plan
1. ENVIRONMENT SETUP: Clone https://github.com/IBM/Evoflux and https://github.com/accenture/mcp-bench. Run `pip install -r requirements.txt`. Create a .env file with OPENROUTER_API_KEY and HF_TOKEN at minimum. Verify the toy benchmark works: `PYTHONPATH=. python3 -m scripts.run_pipeline --benchmark toy --stages search,validate`.

2. UNDERSTAND THE DATA FORMAT: Read the MCP-Bench loader in the repo. A task triple is (natural_language_request, tool_catalog_T, execution_environment_E). The tool catalog lists (server_id, tool_name, schema) tuples. Understand the GraphTemplate / compiled workflow data structures: action stores edits + base graph, candidate stores action + compiled workflow + score + feasibility + execution logs.

3. STAND UP A JUDGE MODEL: The paper uses openai/gpt-oss-20b for small planners. Pick an accessible alternative: gpt-4o-mini via OpenRouter, or a local vLLM-served model on port 8001. The judge must score six sub-dimensions (task fulfillment, grounding, tool appropriateness, parameter accuracy, dependency awareness, parallelism/efficiency) each 1–10, called 5 times with shuffled prompt structure per candidate. Implement or verify the LLMJUDGE wrapper that averages the five runs per sub-dimension and returns a scalar mean.

4. STAND UP A PLANNER MODEL: Start with a small open model, e.g. Qwen2.5-3B-Instruct via vLLM on port 8000, or Llama-3.2-3B via OpenRouter. Implement the PLAN(query, error_context) call: system prompt describes the tool catalog and workflow JSON schema, user prompt contains the task and optional retry error context, assistant output is a GraphTemplate JSON.

5. IMPLEMENT CANDIDATE CONSTRUCTION (BUILDCANDIDATE): Parse GraphTemplate JSON into a graph object. Resolve each node's (server, tool) pair against catalog T—mark infeasible if not found. Validate all required parameters against tool schemas—mark infeasible with schema error. Check graph acyclicity and that all dependency references point to existing nodes. If static checks pass, execute the workflow in environment E by calling MCP servers sequentially in topological order, collecting per-node outputs and errors. Assign score 0 to infeasible candidates.

6. IMPLEMENT THE TYPED EDIT LANGUAGE: Write five mutation functions operating on GraphTemplate objects: (a) tool_swap(node_id, new_server, new_tool) — replaces binding, keeps edges; (b) param_edit(node_id, param_key, new_value_or_upstream_ref) — modifies argument or binds to upstream output; (c) tool_insert(position, server, tool, params, dependencies) — adds evidence/transform step; (d) tool_remove(node_id) — deletes node, reconnects downstream edges if possible; (e) step_reorder(node_id_a, node_id_b) — swaps dependency ordering. Each edit must preserve graph validity or return None if it would break the DAG.

7. IMPLEMENT COLLECT_EVIDENCE: After executing a candidate, walk through execution logs per node and record: did the tool resolve? did parameters satisfy schema? did upstream values exist when consumed? did execution succeed? were outputs empty/malformed? did downstream nodes fail because of this node? Also capture final-answer grounding failures from judge feedback. Return a structured evidence object per node.

8. IMPLEMENT THE ADAPTIVE INTENSITY CONTROLLER: Track G (EMA of squared improvements, decay rho=0.9, epsilon=1e-8). After each new candidate with score f_new against best f_star, compute delta = max((f_new - f_star) / max(|f_star|, eps), 0), then G = 0.9*G + 0.1*delta^2. Compute I = I_min + (I_max - I_min) / (1 + sqrt(G + eps)) with I_min=0.1, I_max=0.7. Draw two independent Bernoulli(I): first controls parent selection (success=uniform, failure=top-quartile tournament), second controls mutation style (success=compound random edits weighted by node error rate, failure=LLM-proposed evidence-conditioned edit).

9. IMPLEMENT META-GUIDANCE: When G < tau_m=0.12 and cooldown counter cd=0, call METAGUIDE(task, best_candidate, worst_3_candidates): send the LLM the task, the current best workflow, and the three weakest population members; ask it to propose a redesigned candidate that can change server paths, introduce missing evidence stages, reorganize dependencies, or simplify the graph. Run this through the same BUILDCANDIDATE pipeline. Set cd = ceil(gamma * B) with gamma=0.2 to prevent repeated redesign calls from dominating the loop.

10. IMPLEMENT ACTION-HASH PRUNING: Assign each candidate an action hash = hash of its (ordered list of (server, tool) tuples across nodes). Bucket the population by action hash into 16 bins. When pruning to K candidates, select round-robin across buckets, preferring higher-scoring candidates within each bucket. This prevents population collapse onto near-duplicate workflows.

11. ASSEMBLE ALGORITHM 2 (FULL EVOLUTIONARY LOOP): Per task: init with retry loop (R=3), run B=5 mutation iterations, track population P (cap K=10), best score f_star, growth G, cooldown cd. At each step: compute I, check meta-guidance trigger, decrement cd, select parent, collect evidence, draw explore flag, mutate with retry, build candidate, update G and f_star, prune population. Return population; evaluation picks highest-scoring candidate.

12. RUN ON MCP-BENCH SEARCH SPLIT: `PYTHONPATH=. python3 -m scripts.run_pipeline --benchmark mcpbench --mcpbench-root /path/to/mcp-bench --complexity single --stages search`. Verify that ValidBest scores are higher than ValidInit scores for your planner.

13. RUN VALIDATION SPLIT: `PYTHONPATH=. python3 -m scripts.run_pipeline --benchmark mcpbench --mcpbench-root /path/to/mcp-bench --complexity single --stages validate`. Check execution feasibility numbers: baseline (~3%) should rise to 17–24% with Evoflux.

14. (OPTIONAL) BUILD TRAINING DATA AND RUN SFT/DPO BASELINES: `python3 -m scripts.build_training_dataset` then `python3 -m scripts.train_sft_dpo`. Compare SFT and SFT+DPO validation scores against Evoflux to reproduce the paper's central finding that training on scarce traces does not reliably beat inference-time search.

## Dependencies
STACK: Python 3.10+, requirements.txt from the repo (likely includes transformers, vllm, torch, pydantic, httpx, python-dotenv, datasets, trl for SFT/DPO). MCP-Bench repo (https://github.com/accenture/mcp-bench) with its 28 live MCP servers. MODELS: Planner — any 1.5B–4B open model (Qwen2.5-3B-Instruct, Llama-3.2-3B, SmolLM3-3B, DeepSeek-R1-Distill-Qwen-1.5B); paper benchmarks also include Qwen3.5-4B, Nemotron-3-Nano-4B, granite-4.0-h-micro. Judge — openai/gpt-oss-20b via API, or claude/sonnet-4.6 for large-model scoring; a gpt-4o-mini substitute is viable for MVP. COMPUTE: For inference-time search with a 3B planner, a single A100 40GB or equivalent is sufficient (vLLM serves at ~50 tok/s). Each query runs B=5 iterations × up to K=10 candidates × execution + 5 judge calls, so expect ~38k–42k tokens per query for small planners. For the SFT/DPO baselines (optional), the 177-example SFT dataset and 624-pair DPO dataset are tiny enough to fine-tune a 3B model in under an hour on one GPU using LoRA (rank=16, alpha=32). DATA: MCP-Bench tasks (provided by the benchmark repo); no additional dataset download required for the core method.

## Limitations / risks
1. LIVE MCP SERVER DEPENDENCY: MCP-Bench requires 28 live external servers (finance, travel, academic search, etc.). These can go down, rate-limit you, or return different results over time, making reproducibility fragile. Some require API keys (NASA_API_KEY, GOOGLE_MAPS_API_KEY). 2. JUDGE MODEL COST: Calling the LLM judge 5 times per candidate, with up to K*B candidates per query, multiplies API costs rapidly. With gpt-4o as judge and 250 eval tasks, cost can reach hundreds of dollars. 3. SMALL PLANNER COLLAPSE: The paper explicitly shows that SFT and SFT+DPO can collapse Qwen3.5-4B below zero-shot performance. Fine-tuning with default hyperparameters on small corpora is high-risk. 4. NARROW BENCHMARK SCOPE: Results are specific to MCP-Bench. Generalization to other tool-use benchmarks (ToolBench, BFCL, custom enterprise APIs) is unvalidated. 5. SEARCH BUDGET SENSITIVITY: B=5 and K=10 are very small. On harder tasks (3-server complexity), feasibility gains may plateau or not materialize if the planner cannot produce a recoverable initial graph (e.g., DeepSeek-R1-Qwen-1.5B stays at 1.27 even after search). 6. HEURISTIC FALLBACK MASKING: When the planner fails all retries, the system silently falls back to keyword-overlap tool selection. This inflates apparent feasibility numbers for weak planners and complicates fair comparison. 7. TOKEN ACCOUNTING INCOMPLETENESS: The reported token-efficiency frontier excludes heuristic fallback token costs, so true per-query costs for weak planners are understated. 8. EVOLUTIONARY SEARCH IS SLOW: Each mutation requires an LLM call, compilation, schema validation, execution, and judge scoring. Wall-clock time per query can be 2–5 minutes even with a fast local server.

## Next steps
1. SCALE THE TRACE BUDGET: The paper's SFT/DPO failure is explicitly scoped to ~177 SFT examples and 624 DPO pairs. Run Evoflux on the full MCP-Bench split to generate 2000–5000 traces and re-run SFT/DPO to test whether distillation becomes competitive at larger scale. 2. APPLY EVOFLUX ON TOP OF TRAINED CHECKPOINTS: The paper notes this as a limitation—Evoflux was only run on base models. Stack Evoflux on an SFT checkpoint to test whether the two are complementary rather than competing. 3. REPLACE THE LLM JUDGE WITH A LEARNED SCORER: The 5-call LLM judge is the main cost driver. Train a lightweight reward model on the judge's outputs from the search split to approximate scores at 1/5 the cost. 4. EXTEND THE TYPED EDIT LANGUAGE: Current edits cover tool swap, param edit, insert, remove, reorder, and validator insert. Add schema-aware parameter synthesis (auto-filling required fields from upstream output types) and server-hopping edits that reconnect a workflow to an alternative MCP server. 5. IMPLEMENT A LEARNED SEARCH CONTROLLER: Replace the hand-tuned intensity formula with a small policy trained via RL on search histories to decide when to explore vs. exploit, when to trigger meta-guidance, and which edit type to apply given evidence. 6. BENCHMARK ON NON-MCP TOOL ENVIRONMENTS: Port the typed workflow DAG representation to ToolBench (16k APIs) or BFCL to test generalization beyond MCP-Bench's 250-tool scope. 7. REDUCE JUDGE CALL COUNT: Explore cheaper scoring proxies—execution success rate, schema validation pass rate, dependency coverage—that can pre-filter candidates before invoking the expensive LLM judge. 8. PARALLELIZE CANDIDATE EVALUATION: The current loop evaluates one mutation per iteration. Batch multiple mutations in parallel (requires async MCP server calls) to keep wall-clock time bounded as B increases.
