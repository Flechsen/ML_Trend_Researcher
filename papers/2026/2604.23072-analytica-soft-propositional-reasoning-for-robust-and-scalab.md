## Metadata

## Why this matters
LLM agents asked to do complex forecasting (financial markets, political predictions, scientific claims) suffer from two core problems: they produce inconsistent answers across runs (high variance) and they systematically miss the right answer (high bias). Analytica attacks both problems formally by decomposing a complex proposition into a tree of simpler sub-claims, grounding leaves with tool-equipped agents, and aggregating scores upward with a linear synthesis rule that is provably noise-resistant. Concrete use cases: (1) automated equity long/short recommendations with explainable reasoning trees, (2) political event forecasting with traceable evidence chains, (3) scientific claim verification in materials science, (4) any 'yes/no' decision that benefits from structured decomposition and parallel evidence gathering.

## Technical idea
Analytica frames complex analysis as Soft Propositional Reasoning (SPR): every claim is assigned a soft truth value in [0,1], and the MSE of that estimate is decomposed into bias and variance. Bias is reduced by recursively decomposing the root proposition into a tree of simpler leaf claims via an Analyzer LLM agent, then grounding each leaf with a tool-equipped Grounder agent (web search, OpenAI Deep Research, or a Jupyter Notebook agent that writes and executes Python). Variance is reduced by synthesizing leaf scores upward through a linear rule: p_parent = beta_0 + sum(beta_j * p_child_j), whose constant partial derivatives give bounded, predictable noise propagation unlike fuzzy-logic alternatives (AND/OR gates whose sensitivity depends on input values). The architecture is massively parallel — all leaves are grounded simultaneously — giving near-linear wall-clock time despite exponential node growth. After a tree is fully grounded, users can edit any node and trigger a fast re-synthesis ('what-if' scenario analysis) without re-running grounders.

## Implementation plan
1. ENVIRONMENT SETUP: Clone https://github.com/chengjunyan1/analytica, create a Python 3.13 conda environment, run 'pip install -e .' and 'analytica setup', then 'pip install -r requirements.txt'. Register the ipykernel: 'python -m ipykernel install --user --name analytica --display-name Python (analytica)'.

2. API KEYS: Obtain and export MY_OPENAI_KEY (required), EXA_API_KEY for web search (required), FMP_API_KEY for financial data, FRED_API_KEY for economic data. Set directory env vars: CACHE_DIR, DB_DIR, TMP_DIR, CKPT_DIR, LOG_DIR. Start with just OpenAI + Exa to keep costs minimal.

3. UNDERSTAND THE CONFIG SYSTEM: Look at configs/test.yaml in the repo. A config specifies the agent type (basic_search | deep_research | jupyter_nb), synthesis rule (vanilla | simple_logic | linear), max_n_leaves (default 10), max_concurrent_prove (default 20), and the LLM model name. Copy test.yaml to configs/my_run.yaml and set grounder to 'basic_search' and synthesis to 'linear' for the MVP.

4. SMOKE TEST WITH A SINGLE PROPOSITION: Run the simple query example from the README verbatim. Pass a Proposition with sentence, context, date, and ckpt_dir. Call system.agent.prove(proposition). Inspect proposition.json to verify you get a p_true float and a report string. Confirm the tree structure is saved in ckpt_dir.

5. UNDERSTAND THE THREE AGENT ROLES: Read analytica/agent/ directory. The Analyzer expands a proposition tree (Algorithm 1 in the paper — iteratively calls LLM to add child nodes until leaf count >= max_n_leaves). The Grounder evaluates leaf nodes (for BasicSearch: call Exa search tool, summarize evidence, output p_true). The Synthesizer aggregates children (for linear rule: prompt LLM to output JSON with beta_0 and beta_j coefficients, then compute weighted sum clamped to [0,1]).

6. IMPLEMENT MINIMAL DATASET: Create a CSV with columns: event_id, description, option_text, ground_truth (1 or 0), date. Start with 20 binary financial market events (e.g., 'Will AAPL be higher in 1 year?') resolved after June 2024. Wrap each row into a Proposition object using the option_text as the sentence and description as context.

7. RUN BATCH EVALUATION: Use the system.evaluate(max_concurrent=10) pattern from the README. Pass your dataset through the system. Results (p_true per proposition, accuracy, cost) are saved to CKPT_DIR. For binary tasks implement the shortcut: evaluate only option_1, set p_option_2 = 1 - p_option_1.

8. IMPLEMENT ACCURACY METRIC: Load ckpt results. For each event, find the option with highest p_true and check if it matches the ground truth option. Compute top-1 accuracy. Also compute Brier Score (mean squared error of predicted probabilities vs. 0/1 labels) for calibration.

9. BASELINE COMPARISON: Run the same 20 events with the grounder-only config (no Analytica decomposition, just direct grounding of the root proposition). Compare accuracy and Brier Score. This replicates the 'BasicSearch' vs '+Analytica-L' rows in Table 2.

10. TEST SYNTHESIS RULE ABLATION: Swap synthesis rule in config between vanilla, simple_logic, and linear. Re-run on the same 20 events. Verify linear rule matches or beats others, reproducing the paper's key finding (Analytica-L > Analytica-V > Analytica-S for stability).

11. JUPYTER NOTEBOOK GROUNDER (ADVANCED): Set grounder to 'jupyter_nb' in config. Ensure FMP_API_KEY and FRED_API_KEY are set. The JupyterSandbox class manages an isolated kernel per leaf. Each grounding session: inject init cell with Proxy API setup, then let agent iteratively write markdown + Python cells (max_proof_steps=20), auto-debug errors (debug_max_retries=5), terminate and compile final report + p_true. This is the cost-effective path to near-Deep-Research accuracy.

12. SCALABILITY TEST: Run Analytica recursively (Analytica2 = each leaf becomes a new root for another Analytica instance). Track wall-clock time vs node count. Confirm near-linear time growth despite exponential node growth, as shown in Table 1.

13. WHAT-IF SCENARIO ANALYSIS: After a tree is fully grounded, manually edit one leaf node's p_true or statement in the saved JSON. Trigger re-synthesis by calling the synthesizer only on the affected branch upward to root. Verify the root p_true updates without re-running grounders.

14. EVALUATION DASHBOARD: Write a simple Python script that loads all ckpt JSON files, computes accuracy/Brier Score/variance across 10 repeated runs on a 20-task subset, and prints a comparison table matching the format of Tables 2-3 in the paper.

## Dependencies
Language models: OpenAI o3-2025-04-16 (primary, reasoning-optimized), or o4-mini for cost-efficiency, or gpt-4.1 for generalist tasks. Any OpenAI-compatible API endpoint works with model name substitution. External APIs: Exa.ai for web search (required), Financial Modeling Prep (FMP) for stock/financial data, FRED for macroeconomic data, Google Trends, optionally Main Street Data and Wolfram Alpha. Python 3.13, conda environment. Core framework: LLLM (https://github.com/chengjunyan1/lllm) installed via 'pip install -e .'. Jupyter kernel for notebook grounding (ipykernel). Key Python packages from requirements.txt (not fully listed in repo but expected: openai, httpx, pyyaml, pandas, numpy, jupyter, nbformat, nbconvert, asyncio). Compute: No GPU needed — all inference is API-based. Parallelism via asyncio (max_concurrent_prove=20 simultaneous leaf groundings). Estimated cost for 736-task benchmark with BasicSearch+Analytica-L: ~$191 (0.26 per task). With JupyterNB+Analytica-L: ~$1.36 per task. With DeepResearch+Analytica-L: ~$14.10 per task. For MVP on 20 tasks with BasicSearch: under $10.

## Limitations / risks
1. INDEPENDENCE ASSUMPTION: The linear synthesis rule assumes child propositions are independent. The Analyzer is prompted to generate independent sub-claims but this is not enforced mathematically — correlated children inflate variance reduction estimates. 2. COEFFICIENT RELIABILITY: The Synthesizer outputs beta coefficients as JSON from an LLM, which can be invalid (wrong range, sum > 1). The system retries up to 3 times but persistent failures degrade accuracy. The paper constrains abs_intercept_max=0.1 but beta_j values are loosely constrained. 3. KNOWLEDGE CUTOFF LEAKAGE: All search results must be filtered to before the event resolution date. The Exa search wrapper enforces a cutoff date but misconfiguration could leak future information, inflating accuracy. 4. JUPYTER SANDBOX SECURITY: The Jupyter grounder executes arbitrary LLM-generated Python code in a local kernel. Without proper sandboxing (containers, network isolation), this is a security risk in production. 5. API COST ESCALATION: Recursive Analytica (depth 3-4) grows nodes exponentially (54x more nodes). At DeepResearch pricing this becomes prohibitively expensive. Cost controls via max_n_leaves are essential. 6. BINARY TASK SHORTCUT: The p_option_2 = 1 - p_option_1 trick only works for mutually exclusive binary tasks. Multi-option tasks require full grounding of each option, multiplying cost. 7. DOMAIN TRANSFER: Results on scientific claims (Matter-of-Fact) show mixed results — GPT-4o-mini performance regressed with Analytica. Older or weaker models may lack the capacity for structured decomposition tasks. 8. EVALS REQUIRE RESOLVED EVENTS: The benchmark only works for events with known ground truth. Building your own dataset requires careful temporal filtering to avoid look-ahead bias. 9. LLLM FRAMEWORK DEPENDENCY: The repo depends on a separate LLLM framework (https://github.com/chengjunyan1/lllm) whose documentation is sparse. Breaking changes in LLLM could break Analytica.

## Next steps
1. GROUNDER ROUTING: Implement adaptive grounder selection — use cheap BasicSearch for straightforward factual leaves and expensive JupyterNB only for quantitative/data-intensive leaves. This maps to the 'hybrid grounder' limitation discussed in Section 6. 2. CORRELATION-AWARE SYNTHESIS: Extend the linear synthesis rule to estimate covariance between child propositions and incorporate it into variance minimization, potentially using techniques from PGMs (Bayesian networks) as sketched in Appendix B. 3. STREAMING UI: Build a real-time web interface showing the proposition tree expanding and leaf scores propagating upward as grounding completes. The existing live demo at analyt1.com provides inspiration. 4. FINE-TUNED ANALYZER: Fine-tune a smaller open-weight model (e.g., Qwen3-80B) specifically on the decomposition task using successful Analytica trees as training data, reducing Analyzer cost while maintaining quality. 5. CALIBRATION LAYER: Add a Platt scaling or isotonic regression calibration step on top of p_true outputs to improve Brier Score, using a held-out validation set of resolved events. 6. MULTI-HYPOTHESIS RANKING: Extend beyond binary tasks to rank N competing hypotheses simultaneously by running Analytica on each and normalizing the softmax of p_true values, enabling richer forecasting tasks. 7. PERSISTENT KNOWLEDGE GRAPH: Cache grounded leaf propositions in a vector database so that recurring sub-claims (e.g., 'Federal Reserve will cut rates') are not re-grounded across different root queries, dramatically reducing cost for related tasks. 8. OPEN-WEIGHT DEPLOYMENT: Validate the full pipeline with a locally-hosted model (e.g., DeepSeek-v3 via Ollama) to eliminate API costs and enable private/on-premise deployment for sensitive financial or scientific use cases.
