# ToolAtlas: Learning Once, Reusing Everywhere with Tool-Side Memory

## Metadata
- arXiv ID: 2607.11126
- Authors: Yue Fang, Zhibang Yang, Fangkai Yang, Xiaoting Qin, Liqun Li, Qingwei Lin, Saravan Rajmohan, Dongmei Zhang
- Published: 2026-07-13
- arXiv link: https://arxiv.org/abs/2607.11126v1
- PDF link: https://arxiv.org/pdf/2607.11126v1
- Code/repo: https://github.com/PuppyKnightUniversity/ToolAtlas

## Why this matters
Most LLM agent systems force every agent to rediscover how tools behave from scratch, wasting compute and producing knowledge that is locked inside one agent's trajectory history. ToolAtlas flips this: it puts a persistent, execution-verified memory graph on the tool provider's side, so any downstream agent can query it. Practical use cases include shared MCP server deployments (GitHub, Stripe, Notion, PostgreSQL) where dozens of heterogeneous agents call the same tools, CI/CD pipelines that need reliable tool-use without per-run exploration, and multi-framework environments where ReAct, CodeAct, and other agents must interoperate over the same tool set. The paper shows up to 21.61% pass@1 improvement and, crucially, the same memory transfers across environment instances and agent frameworks without retraining.

## Technical idea
ToolAtlas builds a three-layer tool memory graph (G_trace, G_cap, G_strat) for an MCP server through two offline phases and uses it at inference via adaptive graph traversal. In Memory Bootstrapping, an LLM proposes k=3 seed tasks per tool (augmented with web-retrieved usage snippets), executes each task N=4 times with a programmatic verifier, then runs Memory Induction to compress the raw rollouts into agent-neutral traces (tool, rationale pairs stripped of action syntax), tool capability entries (affordances U_i, boundaries B_i, co-usage patterns C_i), and strategy nodes (recurring orchestration principles). In Capability Exploration (R=3 rounds), the current G_cap is used as feedback: for each target tool, an LLM proposer generates 3 boundary probe tasks (push the tool until it breaks) and 3 affordance probe tasks (find untested valid uses and compositions), executes them, and folds verified outcomes back into the graph. At inference, Dynamic Memory Traversal starts from the k_r=3 nearest trace nodes by embedding similarity and walks the graph with four operations (ReadTrace, Expand, ReadTool, ReadStrategy) under a budget of B=8 steps, then compresses the gathered context into task-conditioned guidance injected into the user agent's prompt before rollout.

## Implementation plan
1. Clone the repo and install dependencies: git clone https://github.com/PuppyKnightUniversity/ToolAtlas, python -m venv .venv, source .venv/bin/activate, pip install -r requirements.txt. Create .mcp_env with OPENAI_API_KEY and OPENAI_BASE_URL.

2. Verify the smoke test works end-to-end on the included filesystem tasks: python pipeline.py --mcp filesystem --models gpt-4o --task-suite easy --tasks papers/papers_counting --exp-name smoke --timeout 120 --k 1. Confirm results appear under results/smoke/.

3. Understand the data model. Read src/toolatlas/memory/ to understand how G_trace, G_cap, and G_strat are stored (likely as JSON or a lightweight graph structure). Identify the node schemas: trace node q_ij = (task_summary, agent_neutral_trace, task_level_tips), tool node t_i = (affordances, boundaries, co_usage), strategy node = (text, source_queries).

4. Implement or verify Memory Bootstrapping. Confirm src/toolatlas/ contains the seed task proposer. The proposer prompt (from Appendix G) takes target tool name/description/schema plus web search snippets and outputs task_id, summary, description_md, verify_py. Wire web search (e.g., DuckDuckGo or Serper API) to retrieve real-world usage snippets. Set k=3 seed tasks per tool, N=4 rollouts per task.

5. Implement or verify Memory Induction for all three graphs. For G_trace: use the Tool-Trace Induction prompt (Appendix G) to convert N rollouts into agent_neutral_trace + task_level_tips. For G_cap: use the Tool-Capability Induction prompt to extract new_affordance, new_boundaries, new_co_usage entries per tool, citing source queries. For G_strat: use the Tool-Strategy Induction prompt to distill recurring planning patterns with a minimum recurrence threshold. Add semantic edges in G_trace by embedding task summaries with text-embedding-3-small and linking nodes above a cosine similarity threshold.

6. Implement or verify Capability Exploration. For R=3 rounds, sample a K-tool neighborhood from G_cap starting at a target tool and following co-usage edges. Build the proposer context from the tool node's existing U_i, B_i, C_i plus linked traces. Generate 3 boundary tasks and 3 affordance tasks per target tool using the probe proposal prompt. Execute each probe with N=4 rollouts and run Memory Induction to update the graph in place.

7. Implement Dynamic Memory Traversal. At inference, embed the new task query and retrieve k_r=3 nearest trace nodes. Run the navigator loop (budget B=8 steps) using the four operations: ReadTrace, Expand, ReadTool, ReadStrategy. Use the navigator prompt from Appendix G to select operations. Compress the gathered context into a guidance object with seed_candidates, playbook, strategy, and tool_tips sections.

8. Implement the three agent frameworks. ReAct: single function-call actions interleaved with thought steps. CodeAct: Python snippet actions. SwitchAct: expose both interfaces and let the LLM choose per step. The mem-switch, mem-react, and mem-code variants are already stubbed in src/agents/; wire in the guidance object from step 7 as a prepended memory slot in the system or user prompt.

9. Run a minimal end-to-end experiment on the filesystem service. Run Memory Bootstrapping and Capability Exploration: python -m src.memory_split_run (or equivalent) on the train split of tasks/filesystem/. Then run inference on the test split: python pipeline.py --mcp filesystem --models gpt-4o --agent-name mem-switch --task-suite standard --exp-name atlas_test --k 4. Compare pass@1 and pass@4 against the Vanilla baseline.

10. Extend to additional MCP services. To add GitHub, Notion, PostgreSQL, or MCP-Universe services, implement the corresponding MCP service connector under src/mcp_services/<service>/, add tool schemas, write or adapt verifiers, and re-run steps 4-9 for each service. The memory graph is stored per service and is reusable across agents without rerunning.

11. Run ablations to validate each component. Remove G_cap (w/o G_cap), remove G_strat (w/o G_strat), replace Dynamic Traversal with fixed top-k retrieval, remove boundary probes, remove affordance probes. Compare pass@1/pass@4 on the filesystem test split to confirm each component contributes.

12. Evaluate cross-environment and cross-agent transfer. For cross-environment: split environment instances (not tasks) into disjoint train/test sets, build memory on train instances, evaluate on test instances. For cross-agent: build memory with SwitchAct, then run inference with ReAct and CodeAct agents using the same frozen memory graph.

## Dependencies
Python 3.10+. Core packages: openai>=1.109.1 (API access), mcp>=1.27.0 (MCP protocol), tiktoken>=0.12.0 (token counting), httpx>=0.28.1 (async HTTP), numpy>=2.4.4 (embeddings/similarity), playwright>=1.58.0 (browser automation service), psycopg2>=2.9.11 (PostgreSQL service), notion-client>=2.4.0 (Notion service). Embedding model: text-embedding-3-small via OpenAI API (no local GPU needed). LLM backbone: GPT-4o or GPT-4o-mini recommended for MVP (paper uses GPT-5.4/GPT-5.4-mini/Grok-4 which may not be publicly available; substitute with GPT-4o). Compute: CPU-only server is sufficient; paper used 4x Xeon E7-4830 v3 with 503 GB RAM. All LLM inference is remote API calls. Web search for usage snippet retrieval requires a search API key (Serper, Bing, or DuckDuckGo). No local model weights required. No GPU required.

## Limitations / risks
1. API availability: The paper uses GPT-5.4 and Grok-4 which are not publicly accessible; you must substitute GPT-4o and accept potential performance gaps. 2. Cost: Memory Bootstrapping runs k*N=12 rollouts per tool, Capability Exploration runs 6 probe tasks per tool per round for R=3 rounds, totaling ~30 LLM calls per tool before inference. For a service with 94 tools (GitHub), this is ~2800 LLM calls just for memory construction. 3. Verifier quality: Seed task verifiers are LLM-generated Python scripts; bugs in verifiers will silently corrupt the memory graph with wrong pass/fail labels. Manual review of verifiers is essential. 4. Tool API drift: Stored memory becomes stale when tool APIs change; there is no built-in re-verification mechanism. 5. Graph scalability: Semantic edge construction between all trace node pairs is O(n^2) in embedding comparisons; for large services this needs batching or approximate nearest-neighbor indexing. 6. Web search grounding: The seed task proposer relies on web-retrieved usage snippets; if search returns irrelevant results, seed task quality degrades. 7. Environment statefulness: MCP services like Filesystem and PostgreSQL are stateful; rollouts that modify state can interfere with each other if not properly snapshotted and reset. 8. Transfer limits: The paper shows agent-side baselines often degrade on cross-agent transfer; ToolAtlas improves but does not fully close the gap on all services. 9. Single developer scope: Replicating all 8 services across 3 agent frameworks and 3 LLM backbones is a multi-week effort; start with filesystem only.

## Next steps
1. Add automatic memory freshness checks: periodically re-execute a sample of stored probe tasks and flag entries whose outcomes have changed, triggering targeted re-exploration. 2. Implement approximate nearest-neighbor indexing (e.g., FAISS or hnswlib) for G_trace edge construction to scale to services with hundreds of tools. 3. Add a REST API layer so the tool memory graph can be queried by any downstream agent over HTTP, decoupling memory from the Python runtime. 4. Extend to MCP services beyond the eight evaluated: Stripe, Slack, Jira, and other official provider-hosted MCP servers. 5. Experiment with open-weight LLM backbones (Llama-3, Qwen-2.5) as the memory navigator and proposer to reduce API cost. 6. Implement incremental memory updates triggered by production agent failures, so the graph grows from real deployment experience rather than only offline probing. 7. Add a memory governance layer: access control per tool node, versioning of capability entries, and audit logs for who updated which entry and when. 8. Explore multi-hop composition discovery: automatically propose tasks that chain three or more tools based on existing co-usage edges, going beyond pairwise compositions.
