# Memory is Reconstructed, Not Retrieved: Graph Memory for LLM Agents

## Metadata
- arXiv ID: 2606.06036
- Authors: Shuo Ji, Yibo Li, Bryan Hooi
- Published: 2026-06-04

## Why this matters
Most LLM memory systems retrieve context once and hope for the best — a static, brittle approach that fails on multi-hop, temporal, or compositionally complex questions. MRAgent reframes memory access as an active, iterative reconstruction process inspired by cognitive neuroscience: the agent reasons over a structured graph, infers new retrieval cues from intermediate evidence, and prunes irrelevant branches — all before committing to a final answer. Concrete use cases include long-running personal assistants that need to recall and reason across hundreds of sessions, decision-support agents operating over extended interaction histories, and any RAG system where single-shot retrieval repeatedly fails on questions requiring chain-of-thought evidence gathering.

## Technical idea
MRAgent builds a heterogeneous Cue–Tag–Content memory graph from raw dialogue. Construction has three phases: (1) a 'rewrite' LLM call that resolves pronouns, normalises dates, segments episodes, and extracts topics and person facts; (2) a keyword/cue extraction pass that tags each episode with a short associative tag and a set of fine-grained cue keywords; (3) graph assembly that links every (cue, tag) pair to its episode node, plus separate semantic nodes for stable personal facts and topic nodes that aggregate recurring episode themes. At query time, the agent runs an iterative tool-calling loop (up to 8 turns, 10 tool calls per turn). Each turn the LLM inspects the current reconstruction state — active cues/tags and accumulated evidence — selects one or more of seven typed traversal tools (tag→episodes, episode→time, episode→keywords, episode→context, topic→episodes, person→aspects, person+aspect→content), receives candidate nodes, routes/prunes them, and updates its evidence context. The loop terminates when the LLM judges the evidence sufficient or the budget is exhausted. This 'active stateful policy' is provably strictly more expressive than any fixed top-k passive retrieval policy for budget T≥2.

## Implementation plan
1. ENVIRONMENT SETUP: Clone https://github.com/Ji-shuo/MRAgent, create a Python 3.9+ venv, run 'pip install -r requirements.txt' (openai torch numpy tqdm requests regex jsonschema nltk bert_score python-dotenv). Copy .env.example to .env and paste your OpenRouter API key (OPENROUTER_API_KEY=sk-or-v1-...).

2. VERIFY API ACCESS: Open common/config.py and confirm OPENROUTER_URL, the model map (gemini→google/gemini-2.5-flash, claude→anthropic/claude-sonnet-4-5), embedding model (text-embedding-3-large), and judge model (openai/gpt-4o-mini). Do a quick smoke test: python -c "from llm.controller import LLMController; print(LLMController().chat([{'role':'user','content':'ping'}]))".

3. DOWNLOAD BENCHMARK DATA: The repo ships data/dataset_locomo.json and data/dataset_LM.json. Inspect data/get_data.py to understand the conversation+question schema. For a fast first run, identify 2–3 sample IDs from dataset_locomo.json to use with --sample.

4. STAGE 1 — REWRITE (dialogue normalisation): Run 'python run.py --data locomo --model gemini --file dev --sample 0'. This triggers agent/agent.py::rewrite(), which calls the DialogueProcessingPrompt in prompts/prompts.py, resolves pronouns, normalises timestamps, segments episodes, extracts topics and personal_sentences. Output lands at data/locomo/rewrite_gemini/0_rewrite.json. Open it and verify the 'sentence', 'topics', and 'personal_sentences' fields look clean.

5. STAGE 2 — EMBED: The same run call triggers llm/rag_utils.py::get_embeddings() → llm/embeddings.py, which batches all rewritten sentences through text-embedding-3-large and saves a .pkl of normalised 3072-d vectors at data/locomo/embedding/gpt_gemini/0_embedding.pkl. Confirm the file exists and that the vectors are L2-normalised.

6. STAGE 3 — KEYWORD EXTRACTION: Still in the same run, agent/agent.py::extract_keyword() fires the KeywordExtractionPrompt for every episode sentence, producing data/locomo/keyword_gemini/0_keyword.json. Each entry should have sentence_id and keyword list matching the rewrite IDs exactly.

7. STAGE 4 — GRAPH CONSTRUCTION (store): memory/system.py::MemorySystem is populated by agent/agent.py::store(). Inspect MemorySystem to understand the in-memory graph: episode nodes keyed by sentence_id, cue-to-episode edges via tags, semantic nodes for personal facts, topic nodes linked to episode sets. Add a debug print after store() to log node/edge counts.

8. STAGE 5 — ANSWER (tool-calling loop): The agent/agent.py::answer_question() method launches the iterative reconstruction loop via llm/controller.py. The LLM receives the QA+Tool-Use prompt plus the TOOLS schema from agent/tools.py, then calls tools dispatched by agent/tools.py::ToolBridge → memory/controller.py::MemoryController. Trace a single multi-hop question end-to-end by checking the log file under log/locomo/.

9. SINGLE-SAMPLE VALIDATION: Run a single sample to completion: 'python run.py --data locomo --model gemini --file dev --sample 0'. Inspect result/locomo/0_result_gemini_dev.jsonl. Each line should contain gold_answer, prediction, category, evidence_labels, retrieved_support.

10. FULL LOCOMO RUN: Once the single sample looks correct, run all conversations: 'python run.py --data locomo --model gemini --file run1'. Ten worker threads handle QA concurrently. Runs are resumable — already-answered questions are skipped.

11. EVALUATION: Run 'python eval/evaluate_reasoning.py --data locomo --model gemini --file run1 --allfile' to compute F1 and LLM-judge accuracy (gpt-4o-mini grader). Compare Overall J score against the paper's Table 1 baseline of 84.21 for Gemini. Also run 'python eval/evaluate_model.py' for F1/EM breakdown by question type.

12. LONGMEMEVAL RUN (optional, heavier): 'python run.py --data LM --model gemini --file lm1 --ca 0 --lm_batch 10' for multi-session, --ca 1 for single-session-user, --ca 2 for temporal-reasoning. Evaluate with evaluate_reasoning.py pointing at LM data. Target overall J ≈ 72.95 (Table 2).

13. ABLATION CHECKPOINTING: To reproduce ablation variants (CE, CTE without reasoning), modify memory/system.py to skip tag edges (CE) or skip semantic/topic layers (CTE), then re-run. Compare Recall and J on multi-hop questions per Figure 5.

## Dependencies
Python 3.9+. pip packages: openai, torch (CPU build sufficient — used only for L2 norm of embeddings), numpy, tqdm, requests, regex, jsonschema, nltk, bert_score, python-dotenv. External APIs (all via OpenRouter, single key): google/gemini-2.5-flash or anthropic/claude-sonnet-4-5 for reasoning/rewrite/keyword (~8 turns × 10 tool calls per question, plus 3 construction calls per episode); openai/text-embedding-3-large (3072-d) for embedding all episode sentences; openai/gpt-4o-mini for LLM-as-judge evaluation. No local GPU required. Benchmark data: dataset_locomo.json (50 conversations, ~200 QA pairs each) and dataset_LM.json (~500 questions, 115K token histories) are already in the repo. Compute: LoCoMo full run is moderate (hundreds of API calls per sample × 50 samples); budget ~$10–30 in API credits depending on model pricing. Intermediate artifacts (rewrite, keyword, embedding) are cached so reruns are cheap.

## Limitations / risks
1. API cost and latency: Each question triggers up to 8 LLM reasoning turns × 10 tool calls, making deep reconstructions expensive and slow (~minutes per hard question at inference). The paper reports lower token use than baselines, but absolute cost is still non-trivial at scale. 2. Static graph — no forgetting or update: The memory graph grows monotonically; there is no mechanism to update, consolidate, or expire stale facts. Long-lived deployments will accumulate noise. 3. LLM extraction quality: All three construction stages (rewrite, keyword, semantic extraction) rely on LLM JSON output following strict schemas. Malformed outputs, hallucinated IDs, or pronoun-resolution errors during rewrite silently corrupt downstream retrieval. schema.py validators catch some failures but not semantic errors. 4. Single-provider dependency: Everything routes through OpenRouter; a key rotation, rate limit, or model deprecation breaks the entire pipeline. 5. Embedding mismatch: Retrieval uses cosine similarity over text-embedding-3-large vectors for coarse candidate selection (K1=80); if the query and relevant episode use very different surface forms, the coarse retrieval misses the evidence before the LLM even sees it. 6. Multi-turn budget cap: The 8-turn / 10-tool-call cap is a hard ceiling; very deep multi-hop queries may exhaust the budget without finding all evidence. 7. Benchmark leakage risk: dataset_locomo.json ships in the repo, so benchmark data is present locally — care needed to avoid contaminating any fine-tuned LLM used as backbone.

## Next steps
1. Memory maintenance: Add LLM-driven ADD/UPDATE/DELETE operations (Mem0-style) so the graph evolves correctly over time rather than growing unboundedly. 2. Adaptive construction: Replace the static per-episode extraction with a relevance-weighted strategy that spends more extraction budget on information-dense episodes. 3. Learned traversal policy: Replace the greedy LLM action-selection with a lightweight RL-trained router (Search-R1 style) to reduce per-question LLM call count. 4. Persistent graph store: Swap the in-memory MemorySystem for a proper graph DB (Neo4j or Kuzu) to support large-scale, multi-user, long-lived deployments. 5. Cross-session entity resolution: Add coreference resolution across sessions (not just within a session) so that entity cue nodes are correctly merged when the same person appears under different names or pronouns across dialogue files. 6. Streaming / real-time ingestion: Build an incremental pipeline that processes new turns as they arrive rather than batch-processing entire dialogues. 7. Retrieval quality diagnostics: Instrument the tool-calling loop to log which tools contributed to correct vs. incorrect answers, enabling data-driven ablations and tool pruning. 8. Multi-modal memory: Extend the Cue–Tag–Content schema to handle image or audio references in conversations, storing perceptual features alongside text payloads.
