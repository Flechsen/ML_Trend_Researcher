## Why this matters
Most coding agents lock you into one LLM provider, even though different models excel at different tasks (e.g., GLM-5 for algorithms, Qwen3-Max for test generation, Kimi-K2.5 for data science). A smart router that picks the right model per-task can close ~83% of the gap to the theoretical per-task oracle while costing less than always using the best single model. Concrete use cases: (1) a dev shop subscribed to multiple API providers wanting to minimize cost without sacrificing quality, (2) an automated CI/CD pipeline that routes bug-fix tasks to specialized models, (3) any agentic coding assistant that needs to generalize to new task types without retraining.

## Technical idea
Agent-as-a-Router formalizes model routing as a Context-Action-Feedback (C-A-F) loop, which is equivalent to a contextual multi-armed bandit. At each task, the router observes a context (task prompt + metadata + accumulated memory of past decisions), selects an action (which LLM to call), executes the chosen model, verifies the output in a sandbox, receives feedback (performance score + cost), and writes that feedback into a vector memory store keyed by task embeddings. The next decision conditions on this growing history. The concrete instantiation, ACRouter, has three modules: an Orchestrator (fine-tuned Qwen3.5-0.8B + heuristic weighted voting + kNN retrieval from memory), a Verifier (AST parse + sandbox execution + LLM-as-Judge aggregated into a scalar score), and a Memory (online FIFO vector store, cosine kNN, BERT/voyage embeddings, capped at 20K entries). The key empirical finding is that the routing bottleneck is information deficit, not reasoning: giving a vanilla LLM router per-dimension performance statistics yields a +15.3% relative gain, surpassing a hand-crafted heuristic router using the same statistics.

## Implementation plan
1. CLONE AND INSTALL: git clone https://github.com/LanceZPF/agent-as-a-router && cd open-source-acrouter && conda create -n acrouter python=3.11 -y && conda activate acrouter && pip install -r requirements.txt && pip install -e . && python -m unittest discover -s tests to verify the environment.

2. DOWNLOAD BENCHMARK DATA: Run 'python scripts/download_hf_assets.py --minimal --dataset-dir .hf/CodeRouterBench' to pull the pre-computed observation matrix (CodeRouterBench on HuggingFace: Lance1573/CodeRouterBench). This gives you id_results_long.csv (9999 tasks x 8 models), ood176_results_long.csv (176 OOD tasks x 8 models), task JSONL files, and model pricing JSON. No live API calls needed for offline replay.

3. REPRODUCE ID BASELINE (sanity check): Run 'python scripts/run_id.py --output-dir outputs/tmp/id'. Expect AvgPerf~50.14, CumReg~202. This replays cached per-task scores against all routing strategies using the precomputed observation matrix. Inspect outputs/tmp/id/ for per-method breakdown CSVs.

4. REPRODUCE ACROUTER OOD RESULT: Run 'python scripts/run_acrouter_ood176.py --output-dir outputs/tmp/acrouter_ood176'. Expect AvgPerf~73.30, CumReg~15.9 on the 176 OOD tasks. This uses the matrix at raw_matrices/phase2_ood/unified/matrix_acrouter_ood176.json.

5. REPRODUCE ALL BASELINES: Run 'python scripts/run_baselines_ood176.py --output-dir outputs/tmp/baselines_ood176' to get the full Table 3 comparison (LogReg, RouteLLM, kNN, LinUCB, LinTS, single-model baselines) on OOD176.

6. UNDERSTAND THE OBSERVATION MATRIX FORMAT: Open data/matrices/phase1_id/model_pricing.json and data/coderouterbench/id_results_long.csv. Each row is (task_id, model, score, input_tokens, output_tokens, cost_usd). The reward is score - 0.1*cost_usd. The per-task oracle is max_j(R_ij) per row. CumReg = sum over tasks of (oracle_reward_i - chosen_reward_i).

7. BUILD YOUR OWN DIMENSION x MODEL PERFORMANCE MATRIX (optional, if you have API keys): Gather tasks from any coding benchmark (HumanEval, LeetCode, etc.). For each task, call each candidate model via API, run the output through a verifier (unit tests / AST parse), log (task_id, model, score, tokens, cost). Save as JSONL. Use 'python scripts/run_pipeline.py --config configs/eval_pipeline.example.json' pointing to your tasks.jsonl and model_results.jsonl.

8. IMPLEMENT THE ORCHESTRATOR LOGIC (core C-A-F loop): Study src/acrouter_repro/inference.py. The ACRouter class takes candidate_models list, a call_model lambda, and a verify lambda. On each task: (a) embed the task prompt using BGE-large or voyage-code-3, (b) cosine-kNN query the memory store (k=10, threshold=0.5) to retrieve similar past tasks, (c) run weighted voting across: DimensionBest lookup, fine-tuned Qwen3.5-0.8B prediction, LogReg prediction, kNN retrieved model. Argmax of votes = chosen model.

9. IMPLEMENT THE VERIFIER: For executable tasks (code gen, bug fix, algorithm), run the model output in a subprocess with timeout (Docker or local Python sandbox). Score = pass@1 (0 or 1) or partial credit from AST parse. For non-executable tasks (refactoring, understanding), use LLM-as-Judge with a rubric prompt. Aggregate: u_i = sum_k(w_k * s_k) where weights are dimension-specific and sum to 1.

10. IMPLEMENT THE MEMORY MODULE: Use a simple in-memory dict or FAISS index keyed by task embeddings. On each completed loop, write (embedding, chosen_model, score, cost, verifier_trace). Bound at 20K entries FIFO. For retrieval, compute cosine similarity and return top-10 neighbors. Provide the neighbor (model, score) pairs as context to the Orchestrator's voting step.

11. FINETUNE THE QWEN3.5-0.8B ROUTER (optional, for best results): Download base model from HuggingFace (Qwen/Qwen2.5-0.5B or 0.8B). Create training data from the probing set (7080 tasks): input = system_prompt + task_metadata + dimension_stats, label = oracle_best_model for that task. Apply LoRA (r=16, alpha=32, dropout=0.05, attention+MLP layers). Train with standard cross-entropy. Alternatively, download the pre-trained adapter: Lance1573/acrouter-qwen35-08b-router-lora via 'python scripts/download_hf_assets.py --with-router-model --model-dir .hf/router_model'.

12. RUN THE LIVE DEMO (if you have API keys): Set OPENROUTER_API_KEY in env. Edit demos/api_coding_solver/models.example.json with your model endpoints. Run 'python demos/api_coding_solver/solve.py --config demos/api_coding_solver/models.example.json --problem-file demos/api_coding_solver/problems/two_sum.txt'. The demo routes a single problem through the model list until the verifier passes.

13. EVALUATE WITH CUMULATIVE REGRET METRIC: Implement CumReg = sum_i(max_j(R_ij) - R_{i, chosen_j}) where R_ij = score_ij - 0.1*cost_ij. Track this as a running sum over the task stream to see whether your router's regret grows sub-linearly (good) or linearly (bad, equivalent to random). Plot regret vs tasks processed to reproduce Figure 5.

## Dependencies
Python 3.11. Core packages: numpy, pandas, scikit-learn (LogReg, TF-IDF, MLP baselines), faiss-cpu or hnswlib (vector memory), sentence-transformers or FlagEmbedding (BGE-large embeddings, ~560MB model), transformers + peft (Qwen3.5-0.8B LoRA finetuning, ~500MB base model). Optional: voyageai (voyage-code-3 embeddings, API key needed), openai / anthropic / zhipuai SDKs (live model calls). Benchmark data: HuggingFace dataset Lance1573/CodeRouterBench (~few hundred MB CSV + JSONL). Pre-trained router adapter: Lance1573/acrouter-qwen35-08b-router-lora (~50MB LoRA weights). Compute for offline replay: any CPU-only machine, no GPU needed. Compute for LoRA finetuning: 1x GPU with 8GB+ VRAM (H100 used in paper but A10G or RTX 3090 sufficient for 0.8B). Compute for live sandbox execution: Docker installed locally for agentic OOD tasks. No special hardware needed for the ID replay or baseline comparisons.

## Limitations / risks
1. Live API costs: Running all 8 frontier models on 10K tasks costs hundreds of dollars (Claude Opus alone costs ~$87 on the probing set). The offline replay avoids this but limits you to pre-collected scores. 2. Provider cache hit rates are unobservable, so monetary cost estimates are approximate. 3. The Verifier for non-executable tasks (code understanding, refactoring) relies on LLM-as-Judge which introduces noise and additional API cost. 4. The finetuned Qwen3.5-0.8B router requires the probing set distribution to match deployment distribution; it overfits badly on OOD tasks (55.36% vs. random 31.25% on OOD, but static classifiers like LogReg collapse to 19.64%). 5. Memory fills with incorrect routing decisions during early deployment before enough examples accumulate; cold-start performance is weaker. 6. The FIFO memory bound (20K entries) means very long task streams lose early experience; no forgetting strategy is implemented. 7. BGE-large embeddings may not capture fine-grained coding task similarity as well as code-specialized embeddings (voyage-code-3 requires an API key). 8. The agentic OOD evaluation uses a 40-step limit vs. the standard 250-step, so absolute resolved rates are lower than full-budget runs (GPT-5.4 gets 75% at 40 steps but would likely be higher at 250). 9. Model pricing changes over time; all cost figures will drift as providers reprice their APIs.

## Next steps
1. Add new models to the pool without rerunning everything: call each new model on the existing task set (~11K API calls), add their scores to the observation matrix, and rerun the routing evaluation. 2. Add new task dimensions by providing a tasks.jsonl + scoring function and generating model responses. 3. Replace the FIFO memory with a smarter forgetting strategy (e.g., keep failures longer than successes, or cluster-based eviction). 4. Implement the LinUCB/LinTS online bandit baselines from scratch and compare their regret curves against ACRouter on your own task stream. 5. Extend the verifier with domain-specific test harnesses (e.g., pytest for Python, cargo test for Rust) to improve score quality beyond AST parsing. 6. Apply the C-A-F loop to non-coding routing problems: tool selection, API endpoint selection, prompt strategy selection, or thinking-effort allocation as suggested in the paper's Discussion section. 7. Explore parameter-level memory (e.g., continual LoRA updates) instead of kNN retrieval as an alternative Memory implementation. 8. Build a streaming web dashboard that shows cumulative regret in real-time as tasks are processed, to monitor router health in production.
