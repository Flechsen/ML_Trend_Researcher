## Why this matters
Every LLM agent call re-derives the same answer from scratch, paying frontier-model prices each time. A support-ticket router making 10,000 decisions per day at 55 µ$ each spends $550/day on answers it already computed yesterday. AUTO treats this the way compilers treat interpreted code: record the agent's behavior, prove which parts are deterministic, extract them into fast verified programs, and run those for near-zero marginal cost. The practical payoff is 6.4× cost reduction (59 → 2 µ$/item) with 96.9% answer parity, while the system honestly refuses to compile what it cannot verify and deopts to the frontier model for genuine novelty. Use cases include: high-volume classification pipelines (ticket routing, content moderation, intent detection), extraction agents (structured field parsing), policy-routing systems, and any production agent where the same logical inputs appear repeatedly.

## Technical idea
AUTO is a compiler pipeline for LLM agent behavior. A drop-in SDK shim records every effectful span of a live agent run (model calls, tool calls, branch decisions, I/O) into a trace store. A determinism census checks how often each span produces identical output across multiple witnessed runs. Deterministic spans are lowered to a typed task-graph IR whose nodes carry capability effects and uncertainty classes (Deterministic | Probabilistic | Generative). An extraction pass runs enumerative search over a closed 17-operation DSL and, when that fails, LLM-guided CEGIS (propose candidate program → sandbox-execute against all witnessed inputs → reject with counterexample → retry). Non-symbolic residue distills into small decision trees or MLPs. A verification gate does differential replay of every candidate against recorded reality with three-valued verdicts (PASS / FAIL / INCONCLUSIVE); only PASS emits. The emitted artifact is a signed WebAssembly .cbin with a manifest of measured guarantees. At runtime a conformal-calibration guard decides per-input whether to serve the compiled tier-1 path (microseconds, $0 marginal) or deopt to the frontier model tier-0 (paid, slow). Every deopt is captured and fed back into a recompile cycle called the ratchet, so novel inputs are paid for once and then compiled. The key insight is that verification, not synthesis, is the product: any guess that fails differential replay is rejected, and the system measures its own ignorance via calibrated abstention thresholds.

## Implementation plan
1. ENVIRONMENT SETUP: Install Rust 1.96.1 (edition 2024) via rustup with the pinned toolchain in rust-toolchain.toml. Download flatc 25.12.19 exactly from github.com/google/flatbuffers/releases/download/v25.12.19/ (use the appropriate platform binary) and place it at tools/flatc/flatc (or flatc.exe on Windows). Clone the repo: git clone https://github.com/RightNow-AI/auto. Run 'cargo build -p auto-cli' to verify the full workspace compiles. Set OPENAI_API_KEY in your environment for any paid paths.

2. RUN THE TOY-AGENT END-TO-END: Execute 'bash evals/toy-agent/e2e.sh' to walk through the complete pipeline on a trivial fixture. This records the toy agent twice via the Python SDK shim, produces a determinism report, compiles a .cbin through the verification gate, runs it, and exercises all negative paths (wrong implementation blocked, far input abstains, deopt recompiled, tampered artifact refused). Read each step of the script carefully — this is your ground truth for what each CLI command does.

3. UNDERSTAND THE TRACE STORE: Run the record/report steps manually. 'cargo run -p auto-cli -- record --store store.db -- python evals/toy-agent/agent.py' twice, then 'cargo run -p auto-cli -- report --task toy-agent --store store.db'. Inspect store.db with sqlite3 to understand the span schema: task id, input hash, output, cost, latency, error flag. This is the determinism census — a span is witnessed-deterministic when seen ≥2 times with identical output and no errors.

4. WRITE YOUR OWN AGENT WRAPPER: Create a simple Python agent that does a classifiable task (e.g., support ticket routing into 3 categories). Use the SDK shim from sdk/python to wrap your OpenAI calls inside traced closures. Run it 3-5 times against the same input set (20-40 inputs) so every span is witnessed at least twice. Aim for a task where you expect high determinism (classification, extraction, routing — not summarization).

5. WRITE A CONTRACT FILE: Study evals/toy-agent/fake-frontier.contract.toml for the contract format. A contract declares: task id, example input/output pairs (from your recorded reality only), acceptance thresholds (exact match fraction, e.g. 0.95), latency budget, cost budget, and optionally a judge model for generative spans. For a classification task start with exact matching. Run 'cargo run -p auto-cli -- verify --contract your.contract.toml --store store.db' to check the gate without emitting an artifact.

6. COMPILE YOUR FIRST ARTIFACT: Run 'cargo run -p auto-cli -- compile --contract your.contract.toml --store store.db --out your-task.cbin'. For the free (non-LLM) synthesis path this uses enumerative DSL search. If that refuses (outputs are not strings or DSL-expressible), the gate will refuse and tell you why — this is the honest F3 result from the paper. To try LLM-guided CEGIS add '--synth llm' with OPENAI_API_KEY and '--spend-cap-usd 0.10'. Inspect the artifact: 'cargo run -p auto-cli -- inspect your-task.cbin' to read the manifest and verify it reports only measured numbers.

7. RUN THE COMPILED ARTIFACT AND VERIFY PARITY: Run 'cargo run -p auto-cli -- run --artifact your-task.cbin --input YOUR_JSON'. Compare outputs to your reference agent on the same inputs. The manifest will show your parity score from differential replay. This is H3 from the paper.

8. SET UP THE RATCHET LOOP WITH THE DAEMON: Run 'cargo run -p auto-cli -- daemon --store store.db --contract your.contract.toml --out-dir artifacts/ --tier0 frontier:gpt-4o-mini --spend-cap-usd 1.00 --stride 8'. This watches the store, recompiles every K=8 new distinct ingested inputs, and hot-swaps the artifact. In a separate process run the serve command: 'cargo run -p auto-cli -- serve --store store.db --artifact artifacts/latest.cbin'. Now send requests through the HTTP endpoint; novel inputs (guard trips) deopt to tier-0, are captured, and trigger recompilation. This is the ratchet — watch the tier-1 fraction climb.

9. REPRODUCE THE NOVELTY STREAM (H1): Study evals/bench/README.md for the F6 novelty stream setup — 300 items with 3 scheduled distribution shifts. Run the bench script for F1 (ticket triage) and F6 (novelty stream) following the reproduction instructions. This gives you the ratchet curve (Figure 3): cost per item vs stream position across three recompile generations. Compare your curve to the paper's 59 → 2 µ$/item result.

10. REPRODUCE THE FAILURE MODE (TABLE 3): Run the novelty stream in leg-A configuration (loose guard α=0.10, generic frontier deopt reference) and measure tier-1 silent wrong answers. Then run leg-C (faithful deopt reference, α=0.40). This is the most important finding: calibration and reference fidelity, not model capability, decide whether cheap stays correct. Log results to a markdown file matching paper/bench-results.md format with eval-run ids.

11. MEASURE YOUR OWN DETERMINISM CENSUS (H2): Run your custom agent task (step 4) and produce a census table matching Table 1: spans, witnessed, deterministic %, broken down by span type. For classification/extraction tasks you should approach 100%; for any generative span expect much lower. This validates the thesis on your own data.

12. EMBED IN-PROCESS (OPTIONAL LATENCY LADDER): If you need the 18-54 µs latency numbers, build the pyo3 wheel ('cd crates/auto-py && maturin develop') or the napi addon ('cd crates/auto-node && npm install && node-pre-gyp build'). Benchmark the call boundary on your artifact to reproduce the latency ladder from the paper.

## Dependencies
Language/runtime: Rust 1.96.1 (edition 2024, pinned via rust-toolchain.toml), flatc 25.12.19 (exact version, must match flatbuffers crate). Python 3.9+ for the SDK shim and evals (sdk/python). Node.js for the napi embedding (optional). Key Rust crates (from Cargo.toml workspace): wasmtime (wasm sandbox execution), flatbuffers (byte-stable IR serialization), rusqlite (trace store), ed25519-dalek (artifact signing), pyo3 (python embedding), napi (node embedding), scikit-learn via python subprocess (decision tree distillation). External API: OpenAI API key for tier-0 frontier calls (gpt-4o-mini or similar), LLM-guided CEGIS, and judged verification — all fail-closed with explicit spend caps. Compute: No GPU required. All distillation uses scikit-learn decision trees or small MLPs on CPU. Wasm execution is in-process at microsecond latency. The full benchmark cost $0.0621 in API spend. Data: You bring your own agent traces. The benchmark corpora (F1-F6) are in evals/bench/ — 20-40 recorded inputs per family, 56 distinct texts across the 300-item stream. Storage: sqlite for the trace store, local filesystem for the content-addressed artifact registry.

## Limitations / risks
1. DSL expressibility wall: The extraction DSL has exactly 17 operations. If your agent outputs JSON objects, nested structures, or anything non-string, the compiler refuses at all three rungs (this is exactly the F3 result). You cannot compile field extraction in v0 — this is a known honest limitation, not a bug. 2. Lexical guard brittleness: Guards use trigram-Jaccard / hashed-cosine sketches, not semantic embeddings. Paraphrases with disjoint vocabulary trip the guard unnecessarily; lexical cousins with different meaning get admitted silently. The paper measured 125/253 tier-1 answers riding on never-witnessed inputs. Semantic embedding guards are a recorded future target. 3. Loose calibration causes silent failures: A misconfigured guard α (too permissive) produced 48.9% silently wrong tier-1 answers in leg-A. This is the most dangerous failure mode — it looks operationally healthy while serving stale answers. Start with tight α and loosen deliberately. 4. Unfaithful deopt reference breaks recompilation: If the tier-0 reference answers outside the contract vocabulary (e.g., a generic frontier model for a 3-label agent), the verification gate refuses recompilation and the ratchet stalls. The deopt reference must be the same agent (same prompt), not a generic model. 5. Generative residue does not compile at small scale: Free-text summarization (17.5% deterministic) failed its judged threshold at n=40 tickets. You need much more data for generative spans. 6. Synthetic benchmark: All corpora are designed, not production traffic. The 6.4× cost reduction is real on that corpus; your production number depends on your actual repetition rate and task mix. Re-run the protocol on your own recorded traffic. 7. Judged verification inherits judge opinions: LLM-judged equivalence for generative spans is reproducible only up to the judge model named in the manifest. Different judge models can disagree. 8. Single local keypair signing: Current artifact signing uses a local ed25519 keypair, not a proper certificate chain or sigstore. Suitable for local use, not for cross-organization artifact distribution.

## Next steps
1. Semantic embedding guards: Replace trigram-Jaccard with sentence-embedding cosine similarity (e.g., OpenAI ada-002 or a local model). This eliminates the lexical-cousin admission problem and should reduce the calibration exposure measured at 125/253 in the paper. 2. JSON/structured output DSL rung: Add an object constructor and array operations to the 17-operation DSL so that field-extraction tasks (F3, 100% deterministic but currently refused) can compile. This is the highest-value single extension — it unlocks a fully deterministic behavior class. 3. Production traffic re-run: Take a real production agent log, run the census protocol, and measure your actual witnessed-deterministic fraction. The paper's 87.1% is on designed corpora; your number is the experiment that matters for deciding whether to deploy. 4. Sigstore/cosign integration: Replace the local ed25519 keypair with sigstore signing so artifacts carry a verifiable certificate chain useful for supply-chain auditing in multi-team deployments. 5. Generative rung at scale: The summarize family needs a generative rung (fine-tuned small LM) for the 17.5%-deterministic residue. Implement a train/eval/accept loop with a small T5 or similar, gated by the same differential contract. 6. Multi-agent pipeline compilation: The paper sketches multi-span region compilation (each stage its own synthesis problem, inter-stage glue edges as capability boundaries). Implement full F2 pipeline compilation including the priority span under a statistical threshold. 7. Benchmark on additional task families: Add code-generation, RAG pipelines, and tool-chaining tasks to AUTO-BENCH to measure where the 87.1% determinism number holds and where it breaks down. 8. Python/Node SDK auto-instrumentation: Make the SDK shims zero-configuration for the most common agent frameworks (LangChain, LlamaIndex, OpenAI Agents SDK) so recording requires no code changes beyond importing the shim.
