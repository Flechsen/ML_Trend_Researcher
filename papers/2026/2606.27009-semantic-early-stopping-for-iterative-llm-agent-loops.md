# Semantic Early-Stopping for Iterative LLM Agent Loops

## Metadata
- arXiv ID: 2606.27009
- Authors: Sahil Shrivastava
- Published: 2026-06-25

## Why this matters
Most LLM agent loops (Writer→Critic, self-refinement, etc.) terminate via a fixed integer cap — max_iterations. This is wasteful in both directions: easy questions get over-processed (wasted tokens, latency, money) while hard questions get cut off mid-improvement. Semantic Early-Stopping (SHP) replaces that blunt counter with a content-aware rule: halt when the answer stops changing in meaning AND stops improving in quality. Concrete use cases: (1) Any RAG-based QA pipeline using iterative refinement — cut 38% of LLM API spend with no quality loss. (2) Agentic coding assistants that iterate on code reviews. (3) Document summarization loops. (4) Any system where you currently hard-code max_iterations and wonder if you're wasting money. The judge-efficient evaluation protocol is itself valuable: it shows how to fairly compare stopping policies without re-running expensive generation for each policy.

## Technical idea
SHP wraps a Writer→Critic LLM loop with a four-level halt cascade checked each round. First, it embeds every draft using a local sentence-embedding model (BAAI/bge-small-en-v1.5, 384-dim, free, no API) and computes cosine distance between consecutive drafts. When this distance stays below threshold epsilon for k consecutive rounds ('entropy convergence'), the answer has converged in meaning. Second, an Information Score (IS) — a weighted blend of four RAGAS metrics: faithfulness, answer relevancy, context precision, context recall — measures output quality each round. The cascade priority is: (1) Critic says APPROVED → halt immediately; (2) cosine distance < epsilon for k rounds → halt (judge-free, cheap); (3) IS stops increasing → halt (expensive, requires judge call per round); (4) hard failsafe at MAX_ROUNDS → always halt (unconditional, guarantees termination). The key insight from the empirical study: the judge-free entropy_only variant (signals 2+4 only) saves 38% of tokens at statistically indistinguishable quality (delta_IS = -0.004, p=0.81), while the full cascade with per-round judging is actually 129% MORE expensive than baseline for no quality gain. The evaluation uses trajectory replay: generate full trajectories once, replay all policies over cached drafts, cache all judge calls — so comparisons are strictly paired and generation cost is paid once.

## Implementation plan
1. ENVIRONMENT SETUP: Create a Python 3.10+ venv. Install core deps: langchain, langgraph, sentence-transformers, ragas, openai (for OpenAI-compatible endpoints), fastapi, uvicorn, datasets (HuggingFace), scipy, numpy, pandas, matplotlib, python-dotenv. Get an API key for an OpenAI-compatible provider (NVIDIA NIM recommended for free credits; Groq works but rate-limits badly). Set NVIDIA_API_KEY or equivalent in a .env file. Embeddings run locally — no key needed.

2. BUILD THE HALT CASCADE (shp/halting.py): Implement a single pure function shp_should_halt(t, dist_hist, is_hist, feedback, cfg) that checks the four conditions in strict priority order. The failsafe clause (t >= max_rounds) must be unconditional — never skip it regardless of ablation flags. Return a (decision, reason) tuple. This function will be reused by the live loop, post-hoc reason derivation, and offline replay — do not duplicate it.

3. IMPLEMENT SIGNALS: (a) semantic_entropy.py — compute cosine distance using sentence-transformers: d_t = 1 - cosine_sim(embed(x_t), embed(x_{t-1})). Use BAAI/bge-small-en-v1.5 loaded locally. Handle the degenerate zero-norm case by returning 1.0. (b) optimize_score.py — implement IS weight strategies: uniform baseline (equal weights), Shannon entropy weighting (weights proportional to entropy of metric distribution across rounds), and optionally AHP/constrained-LS. (c) ragas_eval.py — wrap RAGAS evaluation (faithfulness, answer_relevancy, context_precision, context_recall) with a disk-based cache keyed by hash(draft_text + question + contexts) to avoid re-judging identical drafts.

4. BUILD RAG-GROUNDED AGENTS (shp/agents.py): Implement Writer and Critic both conditioned on retrieved contexts C. Writer prompt: 'Given question Q and contexts C, write an answer. Previous feedback: {feedback}. Draft a response.' Critic prompt: 'Given question Q, contexts C, and this answer: {draft}, critique it. If it is good enough, reply starting with APPROVED.' This grounding is critical — without it the judge scores faithfulness to contexts the writer never read, invalidating quality numbers.

5. BUILD THE LOOP (shp/agent_workflow.py): Implement the Writer→Critic loop using LangGraph or plain Python. Each round: (a) call Writer to get draft x_t, (b) compute embedding e_t and distance d_t, (c) compute IS_t via cached RAGAS judge, (d) call shp_should_halt — if halts, return (x_t, reason), (e) call Critic to get feedback, (f) increment t. Track token usage via a process-wide token meter (shp/token_meter.py) — log Writer tokens, Critic tokens, and Judge tokens separately. This separation is essential for the operational-vs-evaluation cost analysis.

6. BUILD DATASET (scripts/build_dataset.py): Stream HotpotQA distractor split from HuggingFace (datasets library). Filter to multi-hop hard questions. For each question, form context from gold supporting paragraphs + distractor paragraphs (~4 total). Assign dev/test split deterministically by hashing the example ID (no randomness). Target N=80 total (20 dev, 60 test). Save as JSON lines to backend/data/scenarios.jsonl.

7. IMPLEMENT TRAJECTORY REPLAY (backend/experiments/): (a) trajectory.py — for each scenario, run the full loop to T_max rounds, saving every draft, embedding, IS score, and token count to disk. (b) policies.py — implement each policy as a replay function that takes the cached trajectory and returns (stop_round, reason): fixed_k (baseline at k=6), entropy_only (cosine+failsafe), critic_only, full_shp (all signals), fixed_k1, fixed_k3, random_stop, oracle_is (stop at argmax IS). (c) judge.py — replay-time judging with the same disk cache from step 3. (d) checkpoint.py — resumable cache so interrupted runs continue from where they stopped.

8. IMPLEMENT MACHINE-CHECKED THEORY (shp/theory_checks.py): Write executable tests that verify: (a) Termination — for any dist_hist and is_hist up to T_max, shp_should_halt returns non-continue at t=T_max regardless of all signals. (b) Well-definedness — IS is in [0,1] for valid inputs; weights sum to 1; distance is finite and in [0,2]. (c) Halt-priority consistency — for 1000 random trajectories, the reason returned by the live halt function equals the reason derived by replaying the same function post-hoc. Run with: python -m shp.theory_checks

9. TUNE THRESHOLDS ON DEV SPLIT: Run the full trajectory generation on the 20-question dev split. Try epsilon values in {0.03, 0.06, 0.10} and patience k in {1, 2, 3}. Select the (epsilon, k) pair that maximizes token savings subject to |delta_IS| < 0.02 versus the fixed_k6 baseline. Paper uses epsilon=0.06, k=2.

10. RUN THE FULL EXPERIMENT ON TEST SPLIT: Run trajectory generation on the 60-question frozen test split with the tuned thresholds. Replay all policies. Record per-question: stop_round, operational_tokens (Writer+Critic+judge-if-policy-uses-IS), evaluation_tokens (RAGAS at final round only), final_IS. Never tune on this split.

11. COMPUTE STATISTICS (experiments/stats.py): For each policy vs baseline (fixed_k6): (a) token savings = (tokens_base - tokens_policy) / tokens_base, (b) delta_IS = mean(IS_policy - IS_base) per question, (c) paired t-test and Wilcoxon signed-rank on IS differences, (d) Cohen's d_z, (e) bootstrap 95% CI on delta_IS, (f) TOST non-inferiority test with margin delta=0.02 (tests H0: E[IS_pi - IS_base] <= -delta). All tests are paired across questions.

12. GENERATE FIGURES (experiments/make_figures.py): (a) Efficiency-quality Pareto scatter (x=operational tokens, y=IS, one point per policy), (b) token savings bar chart per policy, (c) mean cosine distance vs round with 95% CI band (to visualize Conjecture 1), (d) results tables (dev split and test split). Save to backend/results/.

13. VALIDATE CONJECTURE 1: Over all test trajectories compute per-step differences d_t - d_{t-1}. Run one-sided Wilcoxon test. Compute OLS slope with bootstrap CI. Report fraction of strictly monotone trajectories. This is a measured conjecture, not a proven theorem — report honestly including null results.

## Limitations / risks
1. JUDGE NOISE: RAGAS quality metrics are themselves produced by an LLM (the same 8B model), making IS a noisy proxy. This is why TOST non-inferiority is not formally certified for entropy_only despite point estimate delta_IS = -0.004 — the confidence interval is too wide. Risk: quality conclusions may not hold with a stronger judge. Mitigation: the paper explicitly uses non-inferiority testing rather than equality claims; replicate with GPT-4 judge if budget allows.

2. BENCHMARK MISMATCH: HotpotQA answers are short and often answerable from a single grounded draft. This is why fixed_k1 (stop after round 1) actually achieves the BEST quality (+0.030 IS). The entropy signal is solving a problem that barely exists on this benchmark. Risk: the 38% token savings claim is real but the 'iteration helps' assumption underlying the motivation does not hold here. Mitigation: the paper acknowledges this honestly; the approach needs validation on long-form generation tasks where iteration genuinely improves quality.

3. FULL CASCADE IS COUNTER-PRODUCTIVE: The full SHP with per-round IS judging costs 129% MORE tokens than baseline for no quality gain. Risk: a naive implementer enables all signals and gets worse results than the baseline. Mitigation: always start with entropy_only (signals 2+4 only); only add the IS signal if you have a very cheap judge.

4. RATE LIMITING: Free-tier providers (Groq) throttle aggressively, making the full experiment take many hours. Mitigation: use NVIDIA NIM for experiments; implement exponential backoff with tenacity; use the checkpoint/resume system so interrupted runs continue.

5. API NON-DETERMINISM: LLM generation is not deterministic across API calls, so trajectory replay is essential — re-running the loop per policy would confound generation noise with policy differences. Risk: if you skip trajectory replay and re-run the loop for each policy, comparisons are not paired and results are meaningless. Mitigation: implement trajectory.py first; never re-run generation per policy.

6. CONTRACTION CLAIM: Do not re-introduce the Banach contraction claim. There is no proven Lipschitz constant below 1 for LLM generation. Conjecture 1 (distances decrease on average) is empirically supported but only ~5% of trajectories are strictly monotone. The termination guarantee comes solely from the unconditional failsafe, not from contraction.

7. ORACLE GAP: The oracle (stop at best round) achieves +0.115 IS over baseline — a massive gap that no online signal captures. This means 'best round selection' is an open problem; entropy stopping solves 'efficient stopping' but not 'optimal stopping'.

## Next steps
1. LONG-FORM BENCHMARK: Replace HotpotQA with a long-form generation task (e.g., multi-step report writing, code generation with tests, essay drafting) where iteration genuinely accumulates quality. The current benchmark under-exercises the loop — the most important validation is showing entropy_only saves tokens AND quality improves with iteration on a task where fixed_k1 does NOT win.

2. ORACLE GAP CLOSURE: The oracle achieves +0.115 IS. Train a lightweight predictor (logistic regression or small MLP) on trajectory features (round index, cosine distance, IS trajectory slope, draft length) to predict which round will have maximum IS — a post-hoc selector at inference time. Even closing 50% of the oracle gap would be a strong result.

3. STRONGER JUDGE: Replicate the experiment with GPT-4 or Claude as the RAGAS judge to certify the TOST non-inferiority that the 8B judge cannot formally certify. This is the most straightforward robustness check.

4. LEARNED STOPPING THRESHOLD: Instead of fixed epsilon and k tuned on dev split, train a small model to predict 'should halt now' from trajectory features. Could approach oracle performance.

5. N-AGENT GENERALIZATION: Extend beyond Writer→Critic to 3+ agent topologies (e.g., Writer→Critic→Fact-checker). The halt cascade is topology-agnostic in principle; verify empirically.

6. EMBEDDING MODEL ABLATION: Test whether bge-small-en-v1.5 (384-dim) vs larger models (bge-large, 1024-dim) affects the quality of the cosine-distance signal. Larger embeddings may detect semantic drift earlier or more accurately.

7. ADAPTIVE EPSILON: Instead of a fixed threshold, compute epsilon adaptively per question type or per early trajectory behavior (e.g., set epsilon = mean(d_2, d_3) * 0.3). This could improve both token savings and quality on diverse question sets.

8. PRODUCTION API: Wrap the entropy_only loop as a drop-in replacement for any max_iterations loop via the FastAPI server already scaffolded in backend/api/app.py. Add a WebSocket endpoint that streams halt decisions in real time for the dashboard.
