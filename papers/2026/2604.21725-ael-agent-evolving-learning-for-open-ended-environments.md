## Why this matters
Most LLM agents reset to zero every episode — they accumulate experience but never learn how to use it better. AEL solves the 'how to use memory' problem, not just the 'what to store' problem. Concrete use cases: (1) A trading bot that gets smarter over hundreds of hourly episodes without retraining weights. (2) Any sequential decision system (web navigation, code repair) where past failures should inform future retrieval strategy. (3) A template for open-ended agents that must self-diagnose degraded performance rather than blindly accumulating logs. The 'less is more' ablation is also practically valuable: it tells builders exactly which complexity to skip.

## Technical idea
AEL is a two-timescale self-improving agent. At the fast timescale, a Thompson Sampling bandit maintains a Beta posterior over a pool of 5 memory retrieval policies (ranging from 'retrieve nothing' to 'aggressive multi-tier recall') and selects one per episode, updating based on a scalar reward clipped to [0,1]. At the slow timescale, every M episodes an LLM receives aggregated trajectories, per-tool accuracy stats, market side information, and the last 3 reflections, then produces a structured causal insight (what conditions caused failures, why specific signals were misleading) plus a regime label. This insight is injected into the planner's prompt — not stored in memory — giving the agent an interpretive frame without polluting the memory store with noisy LLM narratives. Memory itself has three tiers: episodic (raw episode logs), semantic (cross-episode patterns distilled every 10 episodes), and procedural (high-confidence rules injected directly into planner prompts). A composite relevance score combining feature match, quality, recency decay, and tier boost ranks retrieved entries. New retrieval policies are generated by the LLM only when the average bandit reward falls below 0.4, preventing change-on-noise. The critical finding: uniform credit (r_tilde = clip((s+1)/2, 0, 1) applied directly to the bandit posterior) outperforms Shapley-based and LLM-driven credit assignment, and every additional mechanism tested (planner evolution, per-tool selection, cold-start, skill extraction) degrades performance on the 208-episode horizon.

4. STUDY CORE AEL MODULES: Read these files in order: (a) `ael/memory/` — understand the three-tier store and the relevance scoring formula r(q,e) = f_match * quality_factor * recency_decay * tier_boost. (c) The reflection call — locate where the LLM is called with the 4 inputs (episode summaries, tool accuracy stats, market side info, last 3 reflections) and where the causal insight is injected into the planner prompt.

5. RUN THE INCREMENTAL BUILD (MVP core): Run three configs in order and record Sharpe at each step:

   Use seed 42 first, then seeds 123 and 456 to get a 3-seed mean. The jump from step b to step c (~+27%) is the key result.

6. VERIFY THE BANDIT MECHANISM: After step 5, open the log from the AEL run. Confirm that: (a) the 15-bar warm-up runs tool-only predictions before bandit updates begin, (b) the bandit posterior is updated every episode, (c) after every 10 episodes semantic distillation fires, (d) slow-window reflection fires every M episodes (check config for M value). Print the final Beta posteriors for each of the 5 retrieval policies — the highest alpha/(alpha+beta) should correspond to 'compressed' or 'fulldetailed' rather than 'none'.

7. RUN THE ABLATION STUDY (to reproduce Table 3): Run each ablation config from `experiments/configs/incremental/step3_*.yaml` with seeds 42, 123, 456. ## Limitations / risks
1. LLM STOCHASTICITY: Sharpe variance across seeds is real (±0.47 for AEL, much higher for baselines). A single seed result is unreliable — you need at least 3 seeds to see the signal. 2. TEST PERIOD IS 2 WEEKS: The test set (Mar 7-21, 2025) is a specific bear-to-bull transition. The Sharpe 2.13 result is on this specific regime change — it may not generalize to other market periods. 3. CREDIT SYSTEM BUGS: The paper explicitly mentions pre-fix vs post-fix code states for FCC/LLM-FCC credit. Post-fix, FCC collapses to -0.07 Sharpe. Make sure you are running post-fix code (commits b9604f7, f6372a4 are the bug fixes). Check git log before trusting FCC results. 4. DATA LEAKAGE RISK: Market side information (sector returns, volatility regime) is computed from cached price data and given to the reflection LLM but NOT to the predictor. Verify this separation in the code — if side info leaks into prediction, results are invalid. 5. WARM-UP IS CRITICAL: Removing the 15-bar warm-up degrades Sharpe from 2.13 to 1.25. If you modify configs, ensure warm_up=15 is set. 6. MEMORY CAP: Max 500 entries per tier with quality threshold 0.3 for writes. If you run longer horizons, memory may fill with low-quality entries. 7. FINANCIAL DOMAIN ONLY: The paper evaluates exclusively on portfolio allocation. Cross-domain generalization to coding/web benchmarks is stated as future work — do not assume these numbers transfer. 8. REPO STATE: Paper is under review (arXiv 2604.21725); repo may have uncommitted changes between submission and current state. Pin to a specific commit before reproducing.

## Next steps
1. CROSS-DOMAIN VALIDATION: Port AEL to SWE-bench (coding) or WebArena (web navigation) — the framework only needs a context vector and scalar reward signal, so adaptation is mostly in the tool definitions and planner prompts. This is the paper's stated next step. 2. LONGER HORIZONS: Test on a 12-month dataset (vs the current 7-week train). With more data, per-tool Thompson Sampling and planner evolution (currently harmful due to data starvation) may become beneficial — the 'less is more' result is explicitly horizon-dependent. 3. BETTER CREDIT ASSIGNMENT: The paper identifies credit assignment as an open problem. Try RLHF-style preference learning over module outputs as an alternative to Shapley values — it doesn't require enumeration of coalitions and may work with fewer samples. 4. POLICY POOL WARM-START: Instead of 5 hand-designed retrieval policies, use the LLM to generate a diverse initial pool from task descriptions before episode 1 — this is 'cold-start initialization' but applied only to policy diversity, not memory priors (which the ablation shows are harmful). 5. REGIME DETECTION AS A FEATURE: The paper's context vector for LinUCB includes volatility and sector returns. Adding a learned regime classifier (bull/bear/flat) as an explicit feature could improve the bandit's contextual adaptation without the data-starvation problem that kills planner evolution. 6. MULTI-DOMAIN MEMORY TRANSFER: Pre-train the semantic/procedural memory tiers on historical data from similar tasks (transfer learning for the memory store), then fine-tune on the target domain — this could reduce the warm-up requirement.
